    Input Management

    When dealing with input in a game, especially when integrating character movement with physics simulation, it's essential to distinguish between player input (input device data) and character input (simulation-ready data). Additionally, understanding the timing differences between frame updates and fixed updates is crucial for reliable input handling.

    Player Input vs Character Input

    Player Input refers to raw data from input devices like keyboard, mouse, or gamepad. These inputs are read every frame, typically on the main thread during Update().

    Character Input is derived from player input and is used to drive the character’s behavior during the physics-based simulation. Since our character simulation aligns with the physics simulation, this data is processed during FixedUpdate, making it subject to a different time step than the player input.

    Fixed Time step & Input Handling

    What Is a Fixed Time step?

    Unity’s ECS FixedStepSimulationSystemGroup ensures that systems update at consistent intervals (e.g., 60Hz). This is essential for simulations requiring determinism, such as physics or server-side logic.

    However, fixed updates don’t necessarily align 1:1 with regular frame updates:

    • High Frame Rate: You may get multiple regular updates per fixed update.
    • Low Frame Rate: You may get multiple fixed updates per regular update.

    This means:

    • In one rendered frame, there might be zero, one, or many fixed updates.
    • Input captured per-frame might not sync correctly with input consumed per-fixed-update.

    Common Issues with Input in Fixed Updates

    When input is processed in a fixed time step context, two main problems can occur:

    1. Missed Inputs (High Frame Rate)

      • Input events that occur between fixed updates may be lost if they aren't persisted until the next fixed update.

    2. Repeated Inputs (Low Frame Rate)

      • If multiple fixed updates occur between two regular updates, the same input state may be consumed multiple times, causing repeated actions.

    For example, with a jump input:

    • In the first case, the character might fail to jump.
    • In the second, it might jump twice or higher than intended.

    Recommended Input Handling Strategy

    To ensure robust input handling across all frame rates, use the following strategy:

    • Accumulate trigger-based inputs (e.g., button presses) between frames.
    • Consume these inputs once per simulation cycle (usually inside the locomotion state machine).
    • Reset or clear the accumulated trigger after consumption.

    Based on the above types, the package uses source generator to create the following runtime systems:

    PlayerInputManagerSystem

    • Instantiates and initializes the PlayerInputAsset for each player.
    • Calls InitInputAsset, allowing setup of player-specific references (e.g., camera, Unity’s PlayerInput component).
        [Serializable]
    [CreateAssetMenu(fileName = "YourCharacterProfileAsset",
        menuName = "ScriptableObjects/WAYN Games/Your Player Character Input Asset", order = 1)]
    public class YourPlayerCharacterInputAsset : PlayerInputAsset<YourLocomotionInput>
    {
        /// <summary>
        /// Reference to the player's camera used for determining movement directions
        /// and input in relation to the camera's perspective in the game world.
        /// </summary>
        private Camera m_PlayerCamera;
        
        public override void ReadInputsFromAsset(out YourLocomotionInput inputs)
        {
            [...]
        }
        
        public override void InitInputAsset(PlayerGameObject playerGameObject)
        {
            // Get the camera from the player game object and cache it for use when reading inputs.
            m_PlayerCamera = playerGameObject.Instance.Value.GetComponent<PlayerGoAuthoring>().Camera;
        }
    }

    InputReaderSystem

    • Runs every frame (Update) on the main thread.
    • Calls ReadInputsFromAsset(out TInput input) to convert player input into a TInput struct (character input).

    ⚠️ Important: ReadInputsFromAsset() runs on the main thread and must not be Burst-compiled. It may access managed Unity types such as Camera, Input, or MonoBehaviour components. Use this method only for device-to-character input mapping, not for simulation logic.

        [Serializable]
    [CreateAssetMenu(fileName = "YourCharacterProfileAsset",
        menuName = "ScriptableObjects/WAYN Games/Your Player Character Input Asset", order = 1)]
    public class YourPlayerCharacterInputAsset : PlayerInputAsset<YourLocomotionInput>
    {
        /// <summary>
        /// Reference to the player's camera used for determining movement directions
        /// and input in relation to the camera's perspective in the game world.
        /// </summary>
        private Camera m_PlayerCamera;
        
        public override void ReadInputsFromAsset(out YourLocomotionInput inputs)
        {
            Transform cameraTransform = m_PlayerCamera.transform;
            float3 cameraSpaceMovement = cameraTransform.forward * Input.GetAxis("Vertical") +
                                         cameraTransform.right * Input.GetAxis("Horizontal");
            float3 groundSpaceMovement = cameraSpaceMovement - math.projectsafe(cameraSpaceMovement, math.up());
            inputs.Movement = math.normalizesafe(groundSpaceMovement);
            inputs.JumpTrigger = Input.GetKeyDown(KeyCode.Space) ? 1 : 0;
        }
        
        public override void InitInputAsset(PlayerGameObject playerGameObject)
        {
            [...]
        }
    }

    Once the input is read, the system schedules a job that:

    • Loops over all player-controlled characters.
    • Calls ApplyInputsToCharacter() on the character input struct.
    • This is where trigger inputs must be accumulated to ensure consistent behavior across frame rates.
        /// <summary>
    ///     Input commands for the characters' locomotion.
    /// </summary>
    public struct YourLocomotionInput : ICharacterLocomotionInput<YourLocomotionInput>
    {
        /// <summary>
        /// Represents the directional movement input vector for the locomotion system.
        /// This vector is used to define the desired movement direction and magnitude
        /// for a character, typically based on player input, such as keyboard or joystick controls.
        /// </summary>
        public float3 Movement;

        /// <summary>
        /// Represents the input trigger value for initiating a jump action in the locomotion system.
        /// This value is accumulated over time to account for frame rate differences and indicates
        /// whether a jump has been requested by the player.
        /// </summary>
        public float JumpTrigger;

        [...]
        
        public void ApplyInputsToCharacter(in YourLocomotionInput playerInputs)
        {
            Movement = playerInputs.Movement;
            // Trigger inputs need to be accumulated to account for frame rate difference between input capture (update) and consumption (fixed updated)
             JumpTrigger += playerInputs.JumpTrigger;
        }
    }

    To use your input in your state machine, you can access and consume the character input through the LocomotionContext

            internal static void OnIntegrateState(ref YourLocomotionContext ctx)
        {
            // Check if jump was triggered
            if (ctx.InputValue.JumpTrigger > 0)
            {
                // Perform Jump
                // DoJump()
                
                // Consume the jump trigger immediately after jump has been initiated
                ConsumeJumpTrigger(ref ctx);
            }
        }
        private static void ConsumeJumpTrigger(ref YourLocomotionContext ctx)
        {
            YourLocomotionInput value = ctx.InputValue;
            value.JumpTrigger = 0;
            ctx.InputValue = value;
        }
