    Player

    As explained in the package overview, a Player represents the person interacting with the game, typically through an input device (keyboard, gamepad, etc.).

    Following the hybrid approach (integrating GameObjects with Entities), the Player is represented both as a GameObject and as an Entity.

    The Player GameObject

    The Player GameObject must be a prefab with a Player GO Authoring component at its root.

    This component:

    • Holds references to the player’s camera and its camera target (often an empty GameObject under the prefab root).
    • Defines the camera follow strategy for controlled characters.

    Beyond this requirement, the Player GameObject can have any structure and use any input or camera management system you prefer.

    Baking into the Player Entity

    The Player GO Authoring component must be referenced on a Player Character Authoring component, placed in a sub-scene.

    • During baking, the Player GameObject prefab is stored in the PlayerGameObjectPrefab component of the Player entity.
    • At runtime, the PlayerGoManagerSystem instantiates and initializes the prefab.
    • The instantiated GameObject is then accessible through the PlayerGameObjectInstance component on the Player entity.

    Input Assets

    The Player Character Authoring also requires a reference to a PlayerInputAsset<TInput>, which defines how player input is read to construct the character’s TInput.

    This asset can include:

    • Inspector references (e.g., Input Action mappings).
    • Script references, via the method: 
      PlayerInputAsset<TInput>.InitInputAsset(PlayerGameObjectInstance playerGameObject)
      This lets you cache references to MonoBehaviours such as cameras, Cinemachine brains, or PlayerInput components.

    Multiple players can reference the same PlayerInputAsset<TInput>. Like the GameObject prefab, the asset itself is not used at runtime. Instead:

    • The source-generated system #TStates#PlayerInputManagerSystem duplicates the asset.
    • It then calls InitInputAsset(...), passing the Player GameObject instance.
    • Finally, the copy is stored in the Player entity as PlayerInputInstance<TInput>.

    The Player Entity

    In the hybrid approach, the preferred way to interact with MonoBehaviours is through systems. This ensures proper update order between ECS systems and MonoBehaviour updates—crucial for smooth camera tracking and character control.

    Accessing the GameObject

    • Use the PlayerGameObjectInstance component.
    • You can query this component from jobs or Burst functions, but direct GameObject access must occur on the main thread in a non-Burst context.

    Accessing Input

    • Use the PlayerInputInstance<TInput> component.
    • The same restrictions apply: querying can be Burst, but using the input asset must be done on the main thread.

    Controlled Characters

    The Player entity also has a DynamicBuffer<ControlledCharacter> that stores references to the entities of all controlled characters.

    This buffer is used by:

    • CameraRigFollowControlledCharacterSystem — updates the camera target based on the follow strategy defined in the Player GO Authoring component.
    • #TStates#InputReaderSystem — reads input from PlayerInputInstance<TInput> and applies it to all controlled characters by calling: 
      TInput.ApplyInputsToCharacter(TInput)

    Controlled characters can be assigned:

    • In the inspector, via the Player Character Authoring.
    • At runtime, by adding entries to the DynamicBuffer<ControlledCharacter>.
