    State Machine

    The Locomotion package uses a state-machine-based implementation to handle character movement behavior.

    A typical state machine uses three lifecycle methods:

    • OnEnterState — runs when entering a new state.
    • OnUpdateState — runs every frame while in a state.
    • OnExitState — runs when leaving a state.

    For example, when transitioning from a falling state to a grounded state, you might nullify the character’s vertical velocity inside OnEnterState or OnExitState.

    In our character controller, however, many operations must occur in a strict order, regardless of the current state. To handle this, the state machine provides a default execution loop via the interface:

    ILocomotionStateMachine<TStateMachine, TStates, TContext, TProfile, TInput, TStateData>

    This loop runs common state-independent logic automatically. For state-specific behavior, each state must implement the interface:

    ILocomotionState<TContext, TProfile, TInput, TStateData>

    To override the default resolution loop, see Overriding State Machine Behavior.

    Movement and Collision Resolution

    The package relies on the Collide and Slide algorithm to handle character movement and collision resolution.

    Each state defines two core methods:

    • OnIntegrateState — calculates the intended displacement for this frame and updates velocity based on input and external forces (e.g., gravity). Called once per fixed update.
    • OnResolveStateCollisions — applies the movement, performing collision detection and sliding along surfaces when needed.

    How OnResolveStateCollisions works

    1. Read the displacement computed by OnIntegrateState.
    2. Project that displacement onto the last known collision surface.
    3. Perform a physics cast along this path.
    4. If a collision occurs:
      • Move the character to the collision point.
      • Compute the remaining displacement.
    5. Repeat until all displacement is resolved or a maximum number of iterations is reached.

    This iterative collision resolution loop is handled automatically by the state machine. You only need to:

    • Define how sliding is computed.
    • Perform collision checks.

    The package provides helper methods in PhysicsOperations for collision detection and displacement computation.

    Note: State changes can occur between iterations. For example, if the falling state detects ground mid-displacement, the state machine will:

    • Move the character to the ground using the falling state.
    • Switch to the grounded state.
    • Continue resolving the remaining displacement using the grounded state.

    Platform Management

    Characters can also be moved by external sources such as moving platforms.

    During each fixed update, the state machine computes any inherited platform velocity/displacement before calling OnIntegrateState.

    • The PhysicsOperations utilities automatically factor this displacement into collision resolution.
    • You should not add it manually in OnIntegrateState.
    • However, you can project it in OnResolveStateCollisions like you would for player-driven displacement.

    To enable platform tracking, add the PlatformTrackerAuthoring component to the moving object. This automatically computes its velocity each fixed update based on position changes.

    Overlap Resolution

    In many games, collisions are not only caused by the character’s own movement but also by external entities (e.g., moving platforms, falling objects, or other characters).

    To handle this, the state machine performs overlap resolution as the first step of its execution loop:

    • It checks for overlapping colliders using overlap tests.
    • If overlaps exist, it attempts to separate the character iteratively.
    • The process continues until the character is no longer overlapping or a maximum number of iterations is reached.

    This ensures characters can react properly to being pushed or intersected by other moving entities.

    Ground Snapping

    Sometimes characters unintentionally lift off the ground, such as when moving over a slope that suddenly flattens or reverses. This may cause jittery movement or trigger incorrect animation transitions (e.g., briefly switching to a “falling” state).

    To prevent this, the state machine performs ground snapping as its final step:

    • A downward physics cast is performed in the gravity direction.
    • If the ground is within a certain threshold, the character is snapped to it.
    • The last ground contact is stored for use in the next frame’s collision resolution.

    Ground snapping details

    • The default snapping distance is set to the character’s shell width.
    • You can override this by implementing ComputeGroundSnappingDistance in your custom ILocomotionContext<TProfile, TInput, TStateData>.
    • Ground snapping only occurs if the character was grounded earlier in the same fixed update (to avoid sticking during jumps or early snapping when falling).

    Overriding State Machine Behavior

    The package is designed to be extensible.

    • States are implemented as structs with static methods.
    • These are executed in a Burst-compatible context.
    • A source generator automatically creates the boilerplate state machine logic (switch statements, method calls, etc.).

    Because the generated code isn’t directly in your source, it might seem unmodifiable—but you can override the execution loop:

    1. Declare a partial struct with the same name as your state enum, suffixed with StateMachine.

      public partial struct #TStates#StateMachine : IComponentData {}

      You may need to redeclare the IComponentData interface for the compiler to not throw an error.

    2. Implement your own execution logic by overriding:

      ILocomotionStateMachine<TStateMachine, TStates, TContext, TProfile, TInput, TStateData>
    .Execute(ref TStateMachine stateMachine, ref TContext context)

    This lets you fully customize the state machine’s execution loop while keeping the package’s performance and Burst compatibility benefits.
