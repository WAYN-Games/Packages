    Locomotion Package Overview

    The Locomotion Package provides a modular, ECS-based architecture for implementing character control in Unity using state-driven locomotion logic. It cleanly separates gameplay entities—Players, Characters, and Models—and handles input, movement, and visual synchronization in a scalable and maintainable way.

    It integrates seamlessly with Unity's New Input System and Cinemachine, but remains flexible enough to support custom input or camera solutions.

    Core Concepts

    Player vs Character

    The concepts of Player and Character are treated as distinct:

    • Player: Represents the person interacting with the game, typically through an input device (keyboard, gamepad, etc.).
    • Character: An in-game entity that responds to the player's commands, or may be AI-controlled.

    This separation allows for flexible gameplay mechanics, such as:

    • Players switching between characters
    • Players controlling multiple characters simultaneously
    • Characters being controlled by AI logic instead of a player

    Each concept has its own lifecycle and responsibilities within the simulation.

    Character : Simulation vs Presentation

    Every Character is internally divided into two subsystems:

    • Simulation: Responsible for physics, movement, and game logic. This part processes input and updates the character’s state deterministically in the ECS world.
    • Presentation: Responsible for visuals, including animations, mesh rendering, and audio. It visually reflects the state determined by the simulation.

    💡 Why this matters (especially for multiplayer):

    In networked or server-authoritative games, simulation often occurs on the server, while the presentation remains client-side. Decoupling these concerns allows for:

    • Smooth local visuals (even with lag)
    • Efficient bandwidth usage
    • Support for prediction and interpolation
    • Better separation between gameplay code and rendering logic

    GameObjects vs Entities

    While Unity ECS is production-ready, several important features—especially Input, Camera, and Animation—still rely on GameObjects. As such, this package uses a hybrid architecture, where both ECS entities and GameObjects coexist.

    How the hybrid structure works:

    • Player
      • Entity side: Controls logic, such as tracking which characters the player is managing.
      • GameObject side: Handles input (via PlayerInput) and camera (via Cinemachine).
    • Character
      • Locomotor Entity: Simulates physics and processes state-driven movement.
      • Model Entity: Follows the locomotor entity using interpolation, ensuring smooth visuals between fixed updates.
      • GameObject Model: Manages skinned mesh rendering, animation, and audio components.

    This separation allows the simulation systems to remain lightweight, jobified, and Burst-compatible, while the visual systems leverage the strengths of GameObject-based components.
