    Écrire des états personnalisés

    Les états sont les blocs de construction du comportement du personnage. Chaque état définit comment le personnage se déplace, réagit à la physique et transite vers d'autres états. Ce guide couvre l'interface d'état, les méthodes du cycle de vie et les patterns courants.

    L'interface d'état

    Chaque état est une struct non managée qui implémente ILocomotionState :

    [LocomotionState(typeof(MyCharacterStateMachine))]
internal partial struct MyState
{
    public void OnEnterState(ref MyContext ctx) { }

    public void OnIntegrateState(ref MyContext ctx) { }

    public void OnResolveStateCollisions(ref MyContext ctx) { }

    public void OnExitState(ref MyContext ctx) { }
}

    Règles clés :

    • Toutes les méthodes du cycle de vie sont des méthodes d'instance avec des implémentations d'interface par défaut (pattern DIM). Elles sont public void, pas internal static. L'interface ILocomotionState est ajoutée par le générateur de source via le mécanisme partial.
    • Les transitions sont gérées par des structs ILocomotionTransition séparées (voir Transitions ci-dessous) - la struct d'état ne définit pas la logique de transition
    • Le générateur de source dispatche les appels vers la struct d'état correcte - pas de dispatch virtuel à l'exécution
    • Les méthodes d'état s'exécutent à l'intérieur d'un job déjà compilé par Burst, donc l'attribut [BurstCompile] explicite sur les structs et méthodes d'état n'est pas requis
    • Pas de types managés (strings, classes, tableaux) - ce code s'exécute dans des jobs Burst

    Méthodes du cycle de vie

    OnEnterState

    Appelé une fois lorsque la machine à états transite vers cet état. Utilisez-le pour :

    • Initialiser les données spécifiques à l'état (réinitialiser les minuteurs, compteurs)
    • Définir les vélocités initiales
    • Effacer les informations de contact (pour les états aériens)
    public void OnEnterState(ref MyContext ctx)
{
    // Entrée dans un état au sol : annuler tout élan vertical
    ctx.BaseContext.VerticalVelocity = float3.zero;
    ctx.BaseContext.VerticalDisplacement = float3.zero;
}

    OnIntegrateState

    Appelé une fois par frame pendant la phase d'intégration. C'est là que vous calculez le déplacement souhaité du personnage en fonction de l'entrée, de la gravité et des paramètres du profil.

    Votre travail ici est de remplir HorizontalDisplacement, VerticalDisplacement, HorizontalVelocity et VerticalVelocity sur le BaseLocomotionContext.

    La façon la plus simple est de calculer les accélérations et d'appeler UpdateVelocitiesAndComputeDisplacements :

    public void OnIntegrateState(ref MyContext ctx)
{
    ref BaseLocomotionContext baseContext = ref ctx.BaseContext;

    // Calculer la gravité
    float3 gravityAcceleration = baseContext.GravityDirection * baseContext.GravityStrength;

    // Calculer l'accélération de mouvement à partir de l'entrée
    float3 desiredVelocity = ctx.Input.MovementDirection * ctx.Profile.MaxSpeed;
    float accel = ctx.Profile.MaxSpeed / ctx.Profile.TimeToMaxSpeed;
    MathematicsHelpers.ComputeAccelerationToTargetVelocity(
        out float3 moveAcceleration,
        baseContext.HorizontalVelocity,
        desiredVelocity,
        baseContext.TimeStep,
        accel);

    // Appliquer les deux accélérations
    baseContext.UpdateVelocitiesAndComputeDisplacements(
        newHorizontalAcceleration: moveAcceleration,
        newVerticalAcceleration: gravityAcceleration);
}
    Important

    N'ajoutez pas la vélocité de la plateforme ici. L'étape ComputeInheritedDisplacement la gère automatiquement après la résolution de collision. L'ajouter manuellement causerait une double application.

    Le pattern de copie du contexte

    Parce que BaseLocomotionContext est un type valeur sur l'interface du contexte, le modifier nécessite soit :

    Option A : Utiliser un local ref (quand votre contexte expose le champ de stockage) :

    ref BaseLocomotionContext baseContext = ref ctx.BaseContext;
baseContext.HorizontalVelocity = float3.zero;
// Les changements sont écrits via la ref

    Option B : Copier, modifier, réécrire (en passant par l'interface) :

    BaseLocomotionContext baseContext = ctx.BaseContext;
baseContext.HorizontalVelocity = float3.zero;
ctx.BaseContext = baseContext;

    L'exemple utilise l'accès ref parce que la struct de contexte expose BaseContext comme champ public en plus de la propriété de l'interface.

    OnResolveStateCollisions

    Appelé potentiellement plusieurs fois par frame (jusqu'à MaxBounce itérations). Chaque appel représente une itération de l'algorithme collide-and-slide.

    Dans la plupart des cas, votre implémentation délègue simplement à baseContext.Move() :

    public void OnResolveStateCollisions(ref MyContext ctx)
{
    BaseLocomotionContext baseContext = ctx.BaseContext;
    baseContext.Move();
    ctx.BaseContext = baseContext;
}
    Note

    Vous devez utiliser le pattern copier-modifier-réécrire ici parce que Move() modifie la position, le déplacement et les données de contact du contexte en interne.

    Pour les cas d'utilisation avancés, vous pouvez ajouter une logique de réponse de collision personnalisée. Par exemple, appliquer une impulsion aux corps dynamiques que le personnage touche :

    public void OnResolveStateCollisions(ref MyContext ctx)
{
    BaseLocomotionContext baseContext = ctx.BaseContext;
    baseContext.Move();

    // Pousser les corps dynamiques au contact
    if (baseContext.IsInContact &&
        baseContext.LocomotionContact.ValueRO.BodyType == BodyType.Dynamic)
    {
        float3 pushVelocity = baseContext.HorizontalVelocity * 0.5f;
        baseContext.ApplyImpulse(
            ref baseContext,
            baseContext.LocomotionContact.ValueRO.BodyIndex,
            baseContext.LocomotionContact.ValueRO.Entity,
            /* impactorBodyIndex */ ...,
            baseContext.CharacterEntity,
            pushVelocity);
    }

    ctx.BaseContext = baseContext;
}

    OnExitState

    Appelé une fois lorsque la machine à états transite hors de cet état. Utilisez-le pour nettoyer les données spécifiques à l'état :

    public void OnExitState(ref MyContext ctx)
{
    // Réinitialiser le compteur de sauts en quittant l'état de saut
    ctx.StateData.JumpData.JumpsCount = 0;
}

    Transitions

    Les transitions définissent quand et comment le personnage passe entre les états. Le package supporte deux patterns.

    Recommandé : Transitions basées sur les attributs

    Déclarez les transitions comme de petites structs avec [LocomotionTransition]. Le générateur de source les découvre et gère le dispatch, OnExitState et OnEnterState automatiquement.

    [LocomotionTransition]
[EvaluateBefore(typeof(IdleToMoving))]
[EvaluateBefore(typeof(IdleToFalling))]
public struct IdleToJumping : ILocomotionTransition<IdleState, JumpingState>
{
    public bool ShouldTransition(ref MyContext ctx)
        => ctx.BaseContext.IsGroundingContact && ctx.Input.JumpTrigger > 0;
}

    Règles clés :

    1. Le nom de la struct suit la convention {Source}To{Target}
    2. ILocomotionTransition<TFrom, TTo> déclare quels états sont connectés
    3. ShouldTransition retourne true quand la transition doit se déclencher
    4. [EvaluateBefore] / [EvaluateAfter] contrôlent la priorité - la première transition correspondante l'emporte
    5. Le générateur de source appelle OnExitState sur la source et OnEnterState sur la cible automatiquement

    Voir le fichier LocomotionTransitions.cs de l'exemple pour un exemple complet avec 17 transitions.

    Le pattern basé sur les attributs est la seule approche supportée. Le générateur de source découvre toutes les structs [LocomotionTransition], gère les appels exit/enter et rend la priorité des transitions explicite via les attributs [EvaluateBefore]/[EvaluateAfter].

    Consommer les entrées de type déclencheur

    Les entrées de type déclencheur comme les appuis sur le saut doivent être consommées après utilisation. Sinon, le déclencheur se déclenche à chaque frame :

    static void ConsumeJumpTrigger(ref MyContext ctx)
{
    MyInput value = ctx.Input;
    value.JumpTrigger = 0;
    ctx.Input = value;
}

    Appelez ceci dans OnIntegrateState après avoir utilisé la valeur du déclencheur. Le pattern copier-modifier-réécrire est requis parce que Input est un type valeur sur le contexte.

    Réutiliser la logique entre les états

    Les états peuvent partager la logique via des méthodes helper internal static. Par exemple, le JumpingState de l'exemple réutilise FallingState.ComputeGravityAcceleration pour la gravité et MovingState.ComputeHorizontalMovementAcceleration pour le mouvement aérien :

    public void OnIntegrateState(ref MyContext ctx)
{
    // Réutiliser le calcul de gravité de chute
    FallingState.ComputeGravityAcceleration(ref ctx, terminalVelocity, out float3 gravityAccel);

    // Réutiliser l'accélération horizontale de déplacement
    MovingState.ComputeHorizontalMovementAcceleration(
        ref ctx.BaseContext, ctx.Profile.Moving,
        ctx.Input.MovementDirection, ctx.Input.MovementMagnitude,
        out float3 moveAccel);

    ctx.BaseContext.UpdateVelocitiesAndComputeDisplacements(moveAccel, gravityAccel);
}

    Cela évite de dupliquer la logique physique tout en gardant chaque état autonome.

    Patterns d'état courants

    État au sol (Idle / Moving)

    • OnEnterState : Mettre à zéro la vélocité verticale
    • OnIntegrateState : Calculer uniquement l'accélération horizontale
    • Transitions : Vérifier l'entrée de saut, la perte de contact au sol, la pente au-dessus du maximum

    État aérien (Falling / Jumping)

    • OnEnterState : Effacer le contact (LocomotionContact.None), optionnellement hériter la vélocité de la plateforme
    • OnIntegrateState : Appliquer la gravité + contrôle aérien optionnel
    • Transitions : Vérifier le contact au sol, le contact avec un mur

    État de glissade

    • OnIntegrateState : Appliquer la gravité (le personnage glisse le long de la pente)
    • Transitions : Vérifier la perte de contact (vers chute), contact au sol sous la pente maximale (vers mouvement/idle)

    Idées d'états personnalisés

    • Course sur mur : Entrer en sautant sur un mur à grande vitesse ; appliquer la gravité lentement ; permettre le saut mural
    • Dash : Entrer sur déclencheur ; appliquer une vélocité horizontale soudaine ; sortir après le minuteur
    • Nage : Entrer lorsque submergé ; remplacer la gravité par la flottabilité ; permettre le mouvement 3D
    • Escalade : Entrer à une échelle ; désactiver la gravité ; se déplacer le long de l'axe de l'échelle

    Déboguer les états

    Si votre état ne se comporte pas comme prévu :

    1. Vérifiez l'ordre de priorité des transitions - une transition antérieure pourrait se déclencher en premier
    2. Loguez baseContext.LocomotionContact.ValueRO.ContactType pour vérifier la détection de contact
    3. Vérifiez que OnExitState et OnEnterState sont appelés pendant les transitions
    4. Assurez-vous que les entrées de type déclencheur sont accumulées dans ApplyInputsToCharacter et consommées dans OnIntegrateState
    5. Utilisez la visualisation de débogage pour voir les points de contact, les trajectoires de mouvement et les impulsions
