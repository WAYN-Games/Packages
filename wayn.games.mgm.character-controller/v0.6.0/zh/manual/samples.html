    示例

    本页面列出 MGM Character Controller 附带的所有示例，并提供推荐的学习顺序。

    SinglePlayerSample
    教学内容 包含五个状态、转换、输入处理、相机设置和 profile 配置的完整状态机
    难度 初级到中级
    设置时间 5 分钟（参见 快速入门）
    依赖 Unity Input System、Cinemachine

    包含内容

    该示例提供了一个完整的第三人称角色控制器参考实现：

    • 5 个状态：Idle、Moving、Jumping、Falling、Sliding
    • 18 个转换：所有有效的状态到状态转换，带优先级排序
    • 输入处理：带跳跃触发器累积的相机相对移动
    • Profile：可调的速度、加速度、跳跃力、重力和终端速度
    • 预制体：Character（Locomotor + Model）和 Player（Camera + Input）
    • 资产：预配置的 profile 和输入 ScriptableObject

    关键文件 - 学习顺序

    按此顺序阅读示例代码以建立心智模型：

    顺序 文件 教学内容
    1 Runtime/LocomotionStateMachine.cs 锚点结构体 - [LocomotionStateMachine] 属性触发源代码生成
    2 Runtime/Data/LocomotionInput.cs 输入结构体，包含持续字段（Movement）和触发字段（JumpTrigger），以及 ApplyInputsToCharacter 中的累积
    3 Runtime/Data/LocomotionProfile.cs 由子 profile（MovingProfile、FallingProfile、JumpingProfile）组成的 Profile 结构体
    4 Runtime/Data/LocomotionStateData.cs 每状态的持久化数据（JumpData 包含跳跃计数）
    5 Runtime/States/IdleState.cs 最简单的状态 - 进入时将速度归零，处理平台旋转
    6 Runtime/States/MovingState.cs 从输入获取的水平加速度，角色朝移动方向旋转
    7 Runtime/States/FallingState.cs 带终端速度的重力应用，复用 MovingState 的水平加速度
    8 Runtime/States/JumpingState.cs 跳跃力应用、触发器消费、通过 StateData 支持多段跳
    9 Runtime/States/SlidingState.cs 在陡坡上由重力驱动的滑行
    10 Runtime/Transitions/LocomotionTransitions.cs 所有转换作为小结构体，带 [LocomotionTransition] 和 [EvaluateBefore] 用于优先级排序
    11 Runtime/Data/LocomotionContextExtensions.cs 考虑坡度角和平台速度的自定义地面吸附距离
    12 Runtime/Data/LocomotionPlayerInputAsset.cs 相机相对输入读取和 Cinemachine 设置

    尝试修改

    运行示例后，尝试以下修改以加深理解：

    1. 更改移动速度 - 打开 SinglePlayerProfileAsset 并修改 Moving > MaxSpeed。注意 profile 是烘焙的，所以需要重新进入 Play 模式。
    2. 启用二段跳 - 在 profile 中，将 Jumping > MaxJumps 设为 2。JumpingState 已经通过 StateData.JumpData.JumpsCount 支持多段跳。
    3. 禁用空中控制 - 在 JumpingState.OnIntegrateState 中，移除水平加速度代码块。角色将保持其起跳速度，没有空中转向。
    4. 添加调试可视化 - 在子场景中选择角色实体并添加 LocomotionDebugConfigAuthoring 组件。进入 Play 模式并在 Scene 视图中观察接触点和移动路径。

    FlyCamSample
    教学内容 最小化单状态飞行摄像机、无重力移动、实体/摄像机旋转分离、无地面接触的碰撞处理
    难度 初级
    设置时间 5 分钟
    依赖 Unity Input System

    包含内容

    用于在测试场景中自由导航的第一人称飞行摄像机控制器。摄像机在无重力环境中自由浮动，沿障碍物滑行，并使用鼠标视角进行旋转。

    • 1 个状态：Flying（始终活跃 — 不需要转换）
    • 无重力：垂直加速度为零，地面吸附已禁用
    • 旋转分离：实体仅进行偏航旋转；俯仰应用于摄像机 GameObject
    • 碰撞：通过默认的碰撞滑行管线沿墙壁和天花板滑行
    • 输入：WASD/ZQSD 移动，鼠标视角，空格键上升，左 Ctrl 下降

    控制

    输入 动作
    W / Z 前进
    S 后退
    A / Q 左平移
    D 右平移
    鼠标 视角（偏航 + 俯仰）
    空格键 上升
    左 Ctrl 下降

    移动跟随摄像机视角方向 — 向下看并按前进键会使摄像机朝视角方向向下移动。

    工作原理

    架构

    FlyCam 遵循与 SinglePlayerSample 相同的 Player → Character 实体分离，但刻意保持最小化：

    classDiagram
    direction LR

    class PlayerEntity["Player Entity"] {
        CharacterFollower
        LocomotionInputAsset
        PlayerGameObjectPrefab
    }

    class CharacterEntity["Character Entity"] {
        FlyCamStateMachine
        FlyCamInput
        FlyCamStateData ~yaw~
        FlyCamProfile
        LocomotionVelocity
        LocomotionContact
        PhysicsCollider ~capsule~
    }

    class PlayerGo["PlayerGo (GameObject)"] {
        PlayerGoAuthoring
        PlayerInput
    }

    class CameraTarget["CameraTarget (Transform)"] {
        Camera
    }

    PlayerEntity --> CharacterEntity : CharacterFollower
    PlayerEntity ..> PlayerGo : 运行时实例化
    PlayerGo --> CameraTarget : 子 Transform

    旋转：实体与摄像机

    实体仅围绕重力轴旋转（偏航）。这是必要的，因为运动管线在状态运行之前会应用重力对齐，这会与应用于实体的任何俯仰产生冲突。

    俯仰完全在托管（GameObject）侧处理：

    1. FlyCamPlayerInputAsset.ReadInputsFromAsset 从鼠标 Y 轴增量累积俯仰
    2. 将俯仰作为 localRotation 应用于 Camera transform
    3. 将俯仰角传递给 ECS 输入结构体，以便状态计算带俯仰的移动方向
    // 在 FlyCamPlayerInputAsset 中 — 托管侧
m_Pitch = Mathf.Clamp(m_Pitch - look.y * LookSensitivity, -MaxPitch, MaxPitch);
m_CameraTransform.localRotation = Quaternion.Euler(m_Pitch, 0f, 0f);
inputs.Pitch = m_Pitch;

    // 在 FlyCamFlyingState 中 — ECS 侧（实体仅偏航）
quaternion rotation = quaternion.AxisAngle(up, math.radians(stateData.Yaw));
baseContext.CharacterPosition.Rotation = rotation;

// 带俯仰的前进方向用于移动
quaternion pitchedRotation = math.mul(rotation,
    quaternion.AxisAngle(math.right(), math.radians(ctx.Input.Pitch)));
float3 forward = math.mul(pitchedRotation, math.forward());

    无重力移动

    状态从输入计算期望速度，并使用 MathematicsHelpers.ComputeAccelerationToTargetVelocity 实现平滑的加速/减速。不施加任何重力加速度 — 水平和垂直通道都完全由输入驱动。

    期望方向相对于重力轴被分解为水平和垂直分量，然后传递给 UpdateVelocitiesAndComputeDisplacements，重力贡献为零。

    地面吸附已禁用

    FlyCamContext 的 partial 覆盖了 ComputeGroundSnappingDistance 使其返回零，防止地面吸附步骤将摄像机拉到地面：

    public partial struct FlyCamContext
{
    public void ComputeGroundSnappingDistance(in FlyCamProfile profile, out float distance)
    {
        distance = 0f;
    }
}

    过期接触清除

    飞行实体没有转换来清除接触状态（不像地面角色会转换到 Falling）。如果不清除，前一次碰撞的接触会持续存在，导致 Move() 在后续帧中错误地投影位移。

    状态在每次积分开始时清除接触：

    baseContext.LocomotionContact.ValueRW = LocomotionContact.None;

    碰撞滑行步骤会在该帧发生碰撞时设置新的接触。

    示例场景包含：

    1. 子场景：

      • 一个带有 FlyCamCharacterLocomotionAuthoring 的角色实体（胶囊碰撞体） — 分配 profile 资产并配置物理胶囊
      • 一个带有 FlyCamPlayerCharacterAuthoring 的玩家实体 — 引用角色、输入资产和 PlayerGo 预制体

    2. PlayerGo 预制体（运行时实例化）：

      • 带有 Camera 和 CameraTarget 引用的 PlayerGoAuthoring 组件
      • 绑定到 FlyCamActionMap 的 PlayerInput 组件
      • 一个 CameraTarget 子 Transform，Camera 作为其子对象
    Important

    Camera 必须是 CameraTarget Transform 的子对象。CameraRigFollowControlledCharacterSystem 会移动 CameraTarget 以匹配角色实体的位置。Camera 继承该位置并添加自己的本地俯仰旋转。

    关键文件 - 学习顺序

    顺序 文件 教学内容
    1 Runtime/FlyCamStateMachine.cs 最小化状态机锚点 — 单个 [LocomotionStateMachine] 属性
    2 Runtime/Data/FlyCamInput.cs 包含持续（Movement、Pitch）、累积（LookDelta +=）和覆写（VerticalMovement）字段的输入
    3 Runtime/Data/FlyCamProfile.cs 仅包含移动调优和视角灵敏度的简单 profile
    4 Runtime/Data/FlyCamStateData.cs 最小化状态数据 — 仅偏航角
    5 Runtime/Data/FlyCamContextExtensions.cs 覆盖 DIM 默认值以禁用地面吸附
    6 Runtime/States/FlyCamFlyingState.cs 核心逻辑：偏航/俯仰旋转分离、带俯仰的移动方向、无重力移动、过期接触清除
    7 Runtime/Data/FlyCamPlayerInputAsset.cs 托管侧俯仰处理、摄像机旋转、Input System 动作读取

    尝试修改

    1. 调整飞行速度 - 打开 FlyCamProfileAsset 并修改 Moving > MaxSpeed。更高的值使摄像机更快。TimeToMaxSpeed 和 TimeToStop 控制加速和减速的手感。
    2. 更改鼠标灵敏度 - 修改 FlyCamProfileAsset 上的 LookSensitivity（影响偏航）或 FlyCamPlayerInputAsset 上的（影响俯仰）。
    3. 禁用碰撞 - 在 FlyCamCharacterLocomotionAuthoring 检查器中将物理 profile 的碰撞过滤器设置为不与任何层碰撞，或者在 FlyCamFlyingState 中覆盖 OnResolveStateCollisions 直接移动 locomotor 而不进行任何碰撞检查。两种方法都有助于理解碰撞滑行管线的作用。
    4. 添加加速冲刺 - 创建一个 MaxSpeed 更高的第二个 profile 并在运行时切换，或在动作映射中添加冲刺输入。
