    Changelog

    All notable changes to this project will be documented in this file.

    [1.0.0] - 2026-04-03

    Added

    • NUnit test coverage — Unit tests for PostCollisionImpulseComparer; integration tests for state machine execution path using TestCharacter fixture (25 tests across 2 test classes)
    • Samples registration — SinglePlayerSample and FlyCamSample registered in package.json; both now visible in the Unity Package Manager UI

    Removed

    • ProfilerRecorderMap / LocomotionProfilerHUD — Profiler overlay components removed

    Fixed

    • Documentation URLs — Updated documentationUrl and changelogUrl in package.json to v0.6.0
    • Broken doc links and orphaned meta file — Fixed

    [0.3.0] - 2026-03-18

    Added

    • Debug visualization system — Configurable debug display for contact points, movement paths, and collision impulses with per-entity color coding
    • Move() method — Unified movement API with integrated debug visualization
    • Configurable CollisionTolerance — Exposed collision tolerance and BodyType.Character detection on CharacterPhysicsProfile; added ContactType.Step
    • Auto-generated assembly definitions — Editor scaffolding now generates runtime, authoring, and editor asmdef files with proper namespaces and GUID references
    • ILocomotionSystem interface template — New DIM-based locomotion system interface generated by editor scaffolding, replacing inline boilerplate
    • TestCharacter test state machine — Full test state machine under Tests/TestCharacter/ with runtime, authoring, and editor assemblies

    Changed

    • ILocomotionState uses instance-method DIM pattern — State lifecycle methods are now instance methods with default interface implementations instead of static methods
    • CharacterPhysicsProfile moved to ILocomotionContext — Decoupled from ILocomotionProfile so profiles no longer need to carry physics data
    • Base authoring classes renamed — BaseCharacterAuthoring → BaseCharacterLocomotorAuthoring, BaseCharacterBaker → BaseCharacterLocomotorBaker
    • Platform velocity inheritance — Replaced path-tracking system with MotionVelocity-based approach via ComputeInheritedDisplacement
    • Post-collision impulse storage — Replaced NativeQueue<PostCollisionImpulse> with NativeParallelMultiHashMap for better parallel write performance
    • Debug config inspector — Converted to UI Toolkit
    • Locomotion system template simplified — Uses ILocomotionSystem DIM pattern; system struct now delegates to shared interface methods
    • Generated file naming — All scaffolded files now prefixed with state machine name to avoid collisions across multiple state machines
    • Test runtime namespace — Moved to WAYN.Games.MGM.Locomotion.Tests.Runtime

    Fixed

    • Overlap resolution — Burst-compiled resolution for extreme overlap cases and low frame rate scenarios
    • Dynamic body impulse scaling — Corrected impulse magnitude and platform velocity transfer
    • Platform overlap and movement — Fixed characters clipping through or getting stuck on moving platforms
    • Contact system and execute loop ordering — Overhauled contact detection; EvaluateTransition now runs once at the end of the loop
    • Balance behavior — Fixed edge-balancing state detection

    Removed

    • GenerateLocomotionSystem — Removed dead source generator method (superseded by DIM-based template)
    • PhysicsProfileValue on ILocomotionProfile — Moved to context; profiles no longer carry physics configuration

    [0.1.0] - 2025-07-02

    Added

    • Modular locomotion architecture – Introduces a clean separation of concerns by splitting player input, character physics, and the visual model into distinct entities. This design supports both player-controlled and AI-controlled characters, and scales to multiple characters or players in local multiplayer scenarios.
    • State-driven movement system – Character locomotion is governed by a flexible state machine with distinct states (e.g. idle, grounded, jumping, falling, sliding) and smooth automatic transitions between them. Movement behavior is defined by data-driven profiles, making it easy to tune speeds and jump parameters without code changes.
    • Physics-based character controller – Uses physics under the hood for robust movement and collisions. Characters utilize a capsule collider and can step over small obstacles up to a defined height, as well as traverse slopes up to a maximum angle before sliding down steeper inclines. The system handles gravity and falling, inherits movement from moving platforms (so characters ride platforms smoothly), and includes solid collision resolution to prevent getting stuck on edges or corners.
    • Integrated input handling – Built-in support for player input, character movement direction is calculated relative to the camera’s orientation, and an optional “aim” input allows the character to align with the camera direction for intuitive control. Input capture uses the new input system for easily supporting multiple control devices.
    • Camera tracking for characters – A camera target tracking system automatically follows player characters to keep them in view. Integration with the new input system and PlayerInputManager allows natural support for split screen local multiplayer.The camera target follows smoothly the position and rotation in both single-player and local multiplayer scenarios. The solution supports use of Cinemachine cameras.
    • Animation-ready character models – Visual presentation is decoupled from logic by spawning a separate model GameObject for rendering and animation. The locomotion system keeps the model’s transform in sync with the character’s state, enabling developers to hook up animations (e.g. running, jumping) to the character’s movement without affecting the underlying physics simulation.
    • Customizable locomotion profiles – Character movement characteristics are driven by scriptable Locomotion Profile assets. These profiles define parameters like movement speed, acceleration, turning rate, allowed jump count, jump heights, and gravity settings for each character. This allows easy tweaking of locomotion behavior and per-character tuning by adjusting profile values in the editor.
    • Example scenes – Included sample scenes demonstrate the locomotion system in action. A single-player example and a local multiplayer example are provided, showcasing basic player movement, jumping, camera follow, and multiple characters controlled simultaneously.
