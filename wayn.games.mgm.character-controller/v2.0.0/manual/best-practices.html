    Best Practices

    Opinionated recommendations for building with the MGM Character Controller.

    Project Structure

    One state machine per character archetype

    Create separate state machine types for characters with fundamentally different movement (e.g., ground characters vs vehicles vs flying). Do not try to fit all movement into a single state enum.

    Keep states small and focused

    Each state should handle one movement behavior. If a state has many branches ("if grounded do X, else if swimming do Y, else..."), split it into separate states with transitions between them.

    Use sub-profiles for each state

    Group profile parameters by the state that uses them:

    public partial struct MyProfile : ILocomotionProfile
{
    public MovingProfile Moving;
    public FallingProfile Falling;
    public JumpingProfile Jumping;
    public DashProfile Dash;
}

    This keeps the Inspector organized and makes it obvious which parameters affect which state.

    State Logic

    Always use ref access for BaseContext

    Declare a ref local at the top of your state methods to avoid copy-modify-writeback overhead:

    public void OnIntegrateState(ref MyContext ctx)
{
    ref BaseLocomotionContext baseContext = ref ctx.BaseContext;
    // All modifications go through the ref
}

    Exception: OnResolveStateCollisions requires copy-modify-writeback because Move() modifies the context internally.

    Accumulate triggers, overwrite continuous inputs

    In ApplyInputsToCharacter:

    • Continuous (movement direction, aim): overwrite with the latest value
    • Triggers (jump, dash, fire): accumulate with +=

    Always consume triggers in your state logic by resetting them to zero after reading.

    Share computation logic, not state

    Use internal static helper methods to share physics computations across states (e.g., FallingState.ComputeGravityAcceleration). Do not share state data - each state should manage its own initialization and cleanup via OnEnterState / OnExitState.

    Do not add platform velocity manually

    The ComputeInheritedDisplacement step handles platform velocity automatically. Adding it in OnIntegrateState causes double-application. The only exception is OnEnterState for airborne states (e.g., jumping off a platform), where you add PlatformHorizontalVelocity to the character's own velocity to preserve momentum.

    Transitions

    Use the [LocomotionTransition] attribute pattern

    Declare transitions as small structs with [LocomotionTransition] and control priority with [EvaluateBefore] / [EvaluateAfter]:

    [LocomotionTransition(typeof(IdleState), typeof(JumpingState))]
[EvaluateBefore(typeof(IdleToMoving))]
public struct IdleToJumping : IMyTransition
{
    public bool ShouldTransition(ref MyContext ctx)
        => ctx.BaseContext.IsGroundingContact && ctx.Input.JumpTrigger > 0;
}

    This is clearer than inline EvaluateTransitions methods and lets the source generator handle dispatch.

    Put critical transitions first

    Order transitions so the most important ones evaluate first. Typical priority: jump > move > fall > slide. A [EvaluateBefore] on jump over move ensures pressing jump while running works.

    Performance

    Keep all state code Burst-compatible

    • Avoid managed types (strings, classes, arrays) in state logic - states run inside Burst-compiled jobs
    • No managed types (string, class, arrays) in state, context, profile, input, or state data structs

    Use shared colliders

    By default, capsule colliders are shared across entities with identical geometry and filter settings. Only add ForceUniqueColliderAuthoring if you need runtime shape modification per entity.

    Use shared profile blob assets

    All entities referencing the same profile asset share the same blob memory. Design your profiles so that common character types use the same asset.

    Only enable debug visualization when investigating

    Debug components add dynamic buffers and per-frame recording overhead. Remove LocomotionDebugConfigAuthoring from entities you are not actively debugging.

    Debugging

    Use debug visualization before adding logs

    The built-in debug visualization shows contact points, movement paths, and impulses per execution stage. This is faster than logging and works inside Burst jobs.

    Check transition priority when states misbehave

    If a character enters the wrong state, check [EvaluateBefore] ordering. A higher-priority transition may be firing before the one you expect.

    Verify contact types

    Many issues come from incorrect contact classification. Log or inspect ctx.BaseContext.LocomotionContact.ValueRO.ContactType to confirm whether the surface is Ground, Wall, or Ceiling.

    Team Workflow

    One profile asset per character class

    Use ScriptableObject profile assets so designers can tune parameters without touching code. Check these assets into version control.

    Name states and transitions descriptively

    Use the {Source}To{Target} naming convention for transitions (e.g., IdleToJumping, FallingToMoving). This makes the transition graph readable at a glance.

    Test states in isolation

    When adding a new state, start with just one transition into it and one transition out. Verify the core behavior works before wiring up all transitions.
