    Bonnes pratiques

    Recommandations pour construire avec le MGM Character Controller.

    Structure du projet

    Une machine a etats par archetype de personnage

    Creez des types de machines a etats separes pour les personnages ayant des mouvements fondamentalement differents (par exemple, personnages au sol vs vehicules vs vol). N'essayez pas de regrouper tous les mouvements dans un seul enum d'etats.

    Gardez les etats petits et cibles

    Chaque etat doit gerer un seul comportement de mouvement. Si un etat a de nombreuses branches ("si au sol faire X, sinon si en nage faire Y, sinon..."), divisez-le en etats separes avec des transitions entre eux.

    Utilisez des sous-profils pour chaque etat

    Regroupez les parametres de profil par l'etat qui les utilise :

    public partial struct MyProfile : ILocomotionProfile
{
    public MovingProfile Moving;
    public FallingProfile Falling;
    public JumpingProfile Jumping;
    public DashProfile Dash;
}

    Cela garde l'Inspector organise et rend evident quels parametres affectent quel etat.

    Logique d'etat

    Toujours utiliser l'acces ref pour BaseContext

    Declarez un local ref en haut de vos methodes d'etat pour eviter la surcharge de copie-modification-reecriture :

    public void OnIntegrateState(ref MyContext ctx)
{
    ref BaseLocomotionContext baseContext = ref ctx.BaseContext;
    // All modifications go through the ref
}

    Exception : OnResolveStateCollisions necessite la copie-modification-reecriture car Move() modifie le contexte en interne.

    Accumulez les declencheurs, ecrasez les inputs continus

    Dans ApplyInputsToCharacter :

    • Continu (direction de mouvement, visee) : ecrasez avec la derniere valeur
    • Declencheurs (saut, dash, tir) : accumulez avec +=

    Consommez toujours les declencheurs dans votre logique d'etat en les remettant a zero apres lecture.

    Partagez la logique de calcul, pas l'etat

    Utilisez des methodes d'aide internal static pour partager les calculs physiques entre les etats (par exemple, FallingState.ComputeGravityAcceleration). Ne partagez pas les donnees d'etat - chaque etat doit gerer sa propre initialisation et son nettoyage via OnEnterState / OnExitState.

    N'ajoutez pas manuellement la velocite de la plateforme

    L'etape ComputeInheritedDisplacement gere automatiquement la velocite de la plateforme. L'ajouter dans OnIntegrateState cause une double application. La seule exception est OnEnterState pour les etats aeriens (par exemple, sauter depuis une plateforme), ou vous ajoutez PlatformHorizontalVelocity a la velocite propre du personnage pour conserver la quantite de mouvement.

    Transitions

    Utilisez le patron d'attribut [LocomotionTransition]

    Declarez les transitions comme de petites structs avec [LocomotionTransition] et controlez la priorite avec [EvaluateBefore] / [EvaluateAfter] :

    [LocomotionTransition]
[EvaluateBefore(typeof(IdleToMoving))]
public struct IdleToJumping : ILocomotionTransition<IdleState, JumpingState>
{
    public bool ShouldTransition(ref MyContext ctx)
        => ctx.BaseContext.IsGroundingContact && ctx.Input.JumpTrigger > 0;
}

    C'est plus clair que les methodes EvaluateTransitions en ligne et permet au generateur de source de gerer le dispatch.

    Placez les transitions critiques en premier

    Ordonnez les transitions de maniere a ce que les plus importantes soient evaluees en premier. Priorite typique : saut > mouvement > chute > glissement. Un [EvaluateBefore] du saut sur le mouvement assure que presser saut en courant fonctionne.

    Performance

    Gardez tout le code d'etat compatible Burst

    • Evitez les types geres (strings, classes, tableaux) dans la logique d'etat - les etats s'executent dans des jobs compiles avec Burst
    • Pas de types geres (string, class, tableaux) dans les structs d'etat, de contexte, de profil, d'input ou de donnees d'etat
    • Pas de Debug.Log sur le chemin critique - utilisez des gardes [Conditional("ENABLE_UNITY_COLLECTIONS_CHECKS")]

    Utilisez des colliders partages

    Par defaut, les capsule colliders sont partages entre les entites ayant une geometrie et des parametres de filtre identiques. N'ajoutez ForceUniqueColliderAuthoring que si vous avez besoin de modifier la forme a l'execution par entite.

    Utilisez des blob assets de profil partages

    Toutes les entites referencant le meme asset de profil partagent la meme memoire blob. Concevez vos profils de maniere a ce que les types de personnages communs utilisent le meme asset.

    N'activez la visualisation de debogage que lors de l'investigation

    Les composants de debogage ajoutent des tampons dynamiques et une surcharge d'enregistrement par frame. Supprimez LocomotionDebugConfigAuthoring des entites que vous ne deboguez pas activement.

    Debogage

    Utilisez la visualisation de debogage avant d'ajouter des logs

    La visualisation de debogage integree montre les points de contact, les trajectoires de mouvement et les impulsions par etape d'execution. C'est plus rapide que les logs et fonctionne dans les jobs Burst.

    Verifiez la priorite des transitions lorsque les etats se comportent mal

    Si un personnage entre dans le mauvais etat, verifiez l'ordonnancement [EvaluateBefore]. Une transition de priorite superieure peut se declencher avant celle que vous attendez.

    Verifiez les types de contact

    De nombreux problemes proviennent d'une classification incorrecte des contacts. Enregistrez ou inspectez ctx.BaseContext.LocomotionContact.ValueRO.ContactType pour confirmer si la surface est Ground, Wall ou Ceiling.

    Flux de travail en equipe

    Un asset de profil par classe de personnage

    Utilisez des assets de profil ScriptableObject pour que les designers puissent ajuster les parametres sans toucher au code. Committez ces assets dans le controle de version.

    Nommez les etats et transitions de maniere descriptive

    Utilisez la convention de nommage {Source}To{Target} pour les transitions (par exemple, IdleToJumping, FallingToMoving). Cela rend le graphe de transitions lisible d'un coup d'oeil.

    Testez les etats de maniere isolee

    Lorsque vous ajoutez un nouvel etat, commencez avec une seule transition vers celui-ci et une seule transition en sortie. Verifiez que le comportement de base fonctionne avant de connecter toutes les transitions.
