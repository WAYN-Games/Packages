    Depannage

    Cette page liste les problemes courants organises par symptome. Pour chaque probleme : ce que vous observez, pourquoi cela se produit et comment le corriger.

    Le personnage tombe a travers le sol

    Cause : Le sol n'a pas de collider physique visible par le monde ECS.

    Correction :

    1. Selectionnez le GameObject du sol a l'interieur de la sous-scene
    2. Ajoutez un composant Physics Shape (pas un MeshCollider ou BoxCollider legacy)
    3. Verifiez que la sous-scene est chargee (verifiez la Hierarchy pour l'icone de sous-scene)

    Verification : Entrez en mode Play. Ouvrez le Physics Debugger (Window > Analysis > Physics Debugger) et confirmez que le collider du sol est visible.

    Prevention : Utilisez toujours les composants Physics Shape / Physics Body pour les colliders ECS. Les colliders legacy sont ignores par Unity.Physics.

    Le personnage ne repond pas aux inputs

    Cause : L'input ne circule pas du materiel vers le composant ECS.

    Correction - verifiez chaque couche dans l'ordre :

    1. Package Input System : Confirmez que com.unity.inputsystem est installe et que le Active Input Handling est defini sur Input System Package (New) ou Both dans Edit > Project Settings > Player
    2. Entite Player : Verifiez que l'entite Player a l'asset d'input assigne dans le composant Player Character Authoring
    3. Prefab Player : Le GO Prefab Player doit avoir un composant PlayerInput avec les actions d'input assignees
    4. Reference du personnage : L'entite Player doit referencer la bonne entite Character

    Verification : Ajoutez un point d'arret ou un log dans votre methode PlayerInputAsset.ReadInputsFromAsset() pour confirmer qu'elle est appelee a chaque frame.

    Le personnage vibre au sol

    Cause : La position visuelle du personnage se met a jour de maniere discrete a chaque pas fixe, causant un scintillement visible.

    Correction :

    1. Definissez InterpolationMode sur Interpolated dans le Character Physics Profile
    2. Si les vibrations persistent, augmentez legerement CollisionTolerance (essayez 0.01)
    3. Verifiez que le collider du sol est statique (pas un corps dynamique se reconstruisant a chaque frame)

    Verification : Les vibrations devraient disparaitre ou devenir imperceptibles a 60+ FPS.

    Le personnage passe a travers les murs

    Cause : L'algorithme collide-and-slide n'a pas assez d'iterations pour resoudre la collision.

    Correction :

    1. Augmentez MaxBounce dans le Character Physics Profile (essayez 8-10, la valeur par defaut est 6)
    2. Verifiez que la couche de collision du mur est incluse dans le masque CollidesWith du personnage
    3. Si le personnage se deplace tres vite, le deplacement par frame peut depasser la taille du collider - reduisez la vitesse ou augmentez le taux de mise a jour fixe

    Verification : Activez la visualisation de debogage avec les points de contact pour voir si les contacts avec les murs sont detectes.

    Le personnage ne peut pas monter les marches

    Cause : Les conditions de detection des marches ne sont pas remplies.

    Correction :

    1. Augmentez MaxStepHeight dans le Character Physics Profile (doit etre inferieur a la hauteur de la section plate de la capsule : CapsuleHeight - 2 * CapsuleRadius)
    2. Assurez-vous que MaxSlope est suffisamment eleve pour que la face verticale de la marche soit classifiee comme un mur (pas comme le sol)
    3. Verifiez que la geometrie de la marche est propre - les artefacts de maillage incline peuvent tromper le detecteur de marches

    Verification : Activez la visualisation de debogage. Les contacts de mur devraient apparaitre sur la face de la marche, suivis d'un mouvement ascendant reussi.

    Le personnage glisse sur les pentes praticables

    Cause : L'angle de la pente depasse le seuil MaxSlope.

    Correction : Augmentez MaxSlope dans le Character Physics Profile. La valeur par defaut gere la plupart des cas a 45-60 degres.

    Verification : Enregistrez ou inspectez ctx.BaseContext.LocomotionContact.ValueRO.Normal pour calculer l'angle reel de la pente.

    Le saut ne se declenche pas

    Cause : L'input de declencheur de saut est ecrase au lieu d'etre accumule, ou il n'est pas consomme apres utilisation.

    Correction :

    1. Dans votre ICharacterLocomotionInput.ApplyInputsToCharacter, accumulez le declencheur : JumpTrigger += playerInputs.JumpTrigger (pas JumpTrigger = playerInputs.JumpTrigger)
    2. Dans votre PlayerInputAsset.ReadInputsFromAsset, retournez 1f pour le declencheur uniquement sur la frame ou le bouton a ete presse (WasPerformedThisFrame()), pas pendant qu'il est maintenu

    Verification : Ajoutez un log temporaire dans ApplyInputsToCharacter pour voir la valeur accumulee.

    Le saut se declenche a chaque frame

    Cause : Le declencheur de saut n'est pas consomme (remis a zero) apres utilisation.

    Correction : Apres avoir lu la valeur du declencheur dans le OnIntegrateState de votre etat, remettez-le a zero :

    LocomotionInput value = ctx.Input;
value.JumpTrigger = 0;
ctx.Input = value;

    Verification : Le personnage devrait sauter une fois par pression de bouton, pas en continu.

    La transition d'etat ne se declenche jamais

    Cause : Les conditions de la transition ne sont pas remplies, ou une transition de priorite superieure se declenche en premier.

    Correction :

    1. Verifiez les attributs [EvaluateBefore] sur vos transitions - ils controlent l'ordre d'evaluation. La premiere transition correspondante l'emporte.
    2. Verifiez que les conditions dans votre methode ShouldTransition correspondent a l'etat reel a l'execution. Utilisez l'Entity Inspector pour verifier LocomotionContact, LocomotionVelocity et les valeurs de votre composant d'input.
    3. Si vous utilisez le patron EvaluateTransitions en ligne, verifiez l'ordre des verifications if - les verifications anterieures ont une priorite superieure.

    Verification : Ajoutez un log temporaire dans le ShouldTransition de la transition pour voir quelle condition echoue.

    Le personnage est propulse de maniere inattendue

    Cause : Une impulsion externe est appliquee depuis une collision avec un corps dynamique, ou la velocite de la plateforme est appliquee deux fois.

    Correction :

    1. Activez la visualisation de debogage des impulsions pour voir d'ou viennent les forces
    2. Verifiez que vous n'ajoutez pas manuellement PlatformHorizontalVelocity dans OnIntegrateState - le systeme gere automatiquement l'heritage de plateforme dans l'etape ComputeInheritedDisplacement
    3. Si les collisions avec des corps dynamiques injectent trop de velocite, verifiez les valeurs Mass et MassToDynamic dans le Character Physics Profile

    Verification : Desactivez temporairement la gestion des impulsions en surchargeant HandleExternalImpulses avec un no-op sur votre struct de machine a etats.

    Le personnage passe a travers ou se bloque sur les plateformes mobiles

    Cause : La distance d'accrochage au sol est trop petite pour la vitesse de la plateforme.

    Correction :

    1. Surchargez ComputeGroundSnappingDistance (ou la methode partielle OverrideGroundSnappingDistance) sur votre contexte pour prendre en compte la velocite verticale de la plateforme
    2. Consultez le LocomotionContextExtensions.cs de l'exemple pour une implementation de reference qui ajoute platformDistance

    Verification : Activez la visualisation de debogage des points de trajectoire pour voir si le personnage perd le contact au sol sur les plateformes rapides.

    Le generateur de source ne produit pas de code

    Cause : Le generateur de source ne peut pas trouver ou traiter les attributs de votre machine a etats.

    Correction :

    1. Assurez-vous que votre enum d'etats a l'attribut [LocomotionStateMachine]
    2. Assurez-vous que chaque valeur d'enum a un attribut [LocomotionState(typeof(YourState))]
    3. Verifiez que votre projet reference l'assembly Wayn.Games.Locomotion.Runtime
    4. Redemarrez Unity ou reimportez le package - le generateur de source s'execute au moment de la compilation

    Verification : Verifiez la section Roslyn Analyzers dans votre assembly definition pour confirmer que la DLL du generateur de source est referencee.

    Le build compile mais le personnage ne fonctionne pas dans le player

    Cause : Les types generiques utilises par le generateur de source peuvent ne pas etre enregistres pour la compilation AOT.

    Correction :

    1. Assurez-vous d'avoir [assembly: RegisterGenericComponentType(typeof(LocomotionInputAsset<YourInput>))] dans votre assembly
    2. Verifiez les erreurs de compilation Burst dans le log de build - les methodes d'etat doivent etre compatibles Burst (pas de types geres)
    3. Verifiez qu'il n'y a pas de types geres (string, class, tableaux) dans les structs d'etat ou le contexte

    Verification : Compilez avec Development Build active et verifiez le log du Player pour les erreurs Burst ou ECS.

    La visualisation de debogage ne s'affiche pas

    Cause : Les composants de debogage ne sont pas ajoutes ou la vue Scene n'est pas visible.

    Correction :

    1. Selectionnez le GameObject d'authoring de l'entite personnage dans la sous-scene
    2. Ajoutez un composant LocomotionDebugConfigAuthoring
    3. Configurez les visualisations a activer dans l'Inspector
    4. Entrez en mode Play et passez a la vue Scene (les gizmos de debogage ne s'affichent pas dans la vue Game)

    Verification : Les donnees de debogage sont cuites dans un blob asset, les modifications necessitent donc de relancer le mode Play.
