    Integration: Animation

    Characters driven by the MGM Character Controller use a separate Model entity with an attached GameObject for rendering and animation. The Animator component lives on this GameObject, not on the ECS entity.

    Architecture

    Locomotor Entity (ECS) → CharacterFollower → Model Entity (ECS) → GameObject (Animator)
    1. The Locomotor entity runs state machine logic in Burst jobs
    2. The Model entity follows the Locomotor via CharacterFollowerSystem
    3. The Model's GameObject gets its transform synced via EntityGoTransformSyncSystem
    4. A MonoBehaviour on the GameObject reads entity state and drives the Animator

    Setup

    1. Create the model GameObject

    In the sub-scene:

    1. Create a child GameObject with your character mesh and Animator
    2. Add a CharacterFollowerAuthoring component
    3. Set the Locomotor field to reference the character's Locomotor authoring component

    2. Create an animation bridge MonoBehaviour

    Since Burst jobs cannot touch the Animator, you need a MonoBehaviour bridge that reads ECS state and sets Animator parameters:

    public class LocomotionAnimationBridge : MonoBehaviour
{
    Animator _animator;
    EntityGameObject _entityLink;

    // Animator parameter hashes
    static readonly int SpeedHash = Animator.StringToHash("Speed");
    static readonly int IsGroundedHash = Animator.StringToHash("IsGrounded");
    static readonly int IsJumpingHash = Animator.StringToHash("IsJumping");

    void Awake()
    {
        _animator = GetComponent<Animator>();
        _entityLink = GetComponent<EntityGameObject>();
    }

    void LateUpdate()
    {
        if (_entityLink == null || !_entityLink.IsValid) return;

        // Read entity data from the ECS world
        var entityManager = World.DefaultGameObjectInjectionWorld.EntityManager;
        var entity = _entityLink.Entity;

        // Read velocity for speed parameter
        if (entityManager.HasComponent<LocomotionVelocity>(entity))
        {
            var velocity = entityManager.GetComponentData<LocomotionVelocity>(entity);
            _animator.SetFloat(SpeedHash, math.length(velocity.Value));
        }

        // Read contact for grounded state
        if (entityManager.HasComponent<LocomotionContact>(entity))
        {
            var contact = entityManager.GetComponentData<LocomotionContact>(entity);
            _animator.SetBool(IsGroundedHash,
                contact.ContactType == ContactType.Ground);
        }
    }
}

    3. Attach to the model GameObject

    Add LocomotionAnimationBridge to the same GameObject that has the Animator. The EntityGameObject component is added automatically by the hybrid layer during initialization.

    State-Aware Animation

    To read the current state machine state, you need access to the generated state machine component. Since this is a generic type, you can either:

    Option A: Use a managed component or shared data structure to expose the current state to MonoBehaviours.

    Option B: Create a system that writes animation-relevant data to a dedicated "animation data" component readable from the bridge.

    Option B is recommended because it keeps the main thread work minimal:

    public struct AnimationData : IComponentData
{
    public float Speed;
    public bool IsGrounded;
    public bool IsJumping;
    public bool IsFalling;
}

[UpdateAfter(typeof(MyLocomotionSystem))]
partial struct AnimationDataSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        foreach (var (animData, velocity, contact) in
            SystemAPI.Query<RefRW<AnimationData>,
                RefRO<LocomotionVelocity>,
                RefRO<LocomotionContact>>())
        {
            animData.ValueRW.Speed = math.length(velocity.ValueRO.Value);
            animData.ValueRW.IsGrounded =
                contact.ValueRO.ContactType == ContactType.Ground;
        }
    }
}

    Known Caveats

    • Animation is always one frame behind physics because the Animator runs after ECS syncs the transform
    • Root motion is not supported - the Locomotor entity drives position, not the Animator
    • Blending between states should be handled by the Animator Controller, not by the locomotion state machine
