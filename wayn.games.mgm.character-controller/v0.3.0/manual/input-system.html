    Input System

    The MGM Character Controller separates input reading from input consumption. This design supports multiple input sources (keyboard, gamepad, AI, network), handles frame rate differences between Update and FixedUpdate, and works cleanly with Unity's new Input System.

    Architecture

    Input flows through three layers:

    flowchart TD
    A["<b>PlayerInputAsset</b><br/>Reads hardware input"]
    B["<b>ICharacterLocomotionInput</b><br/>Accumulates input per frame"]
    C["<b>State logic</b><br/>Reads TInput on context"]

    A -- "ReadInputsFromAsset(out TInput)" --> B
    B -- "(generated InputReaderSystem)" --> C

    A -.- A1["MonoBehaviour layer (Update rate)<br/>e.g., keyboard, gamepad, touch"]
    B -.- B1["ECS component (Update rate)<br/>ApplyInputsToCharacter()"]
    C -.- C1["Burst job (FixedUpdate rate)<br/>OnIntegrateState, EvaluateTransitions"]

    Why Three Layers?

    Frame rate mismatch: Unity's Update loop runs at variable rate (e.g., 60-144 fps), but FixedUpdate runs at a fixed rate (e.g., 50 fps). If a player presses Jump during a frame where no FixedUpdate runs, that press must not be lost. The accumulation layer ensures trigger inputs survive until the next fixed step.

    Burst compatibility: PlayerInputAsset is a ScriptableObject that touches MonoBehaviour APIs (Camera, PlayerInput, Input System). These cannot run inside Burst jobs. The separation allows the managed-code layer to read input and the Burst-compiled layer to consume it.

    Multiple input sources: By abstracting input behind ICharacterLocomotionInput, you can swap the input source without changing any state logic. A keyboard player, a gamepad player, and an AI agent all write to the same TInput component.

    PlayerInputAsset

    PlayerInputAsset<TInput> is an abstract ScriptableObject that you subclass to define how hardware input maps to your input struct.

    [CreateAssetMenu(menuName = "My Game/Player Input Asset")]
public class MyPlayerInputAsset : PlayerInputAsset<MyInput>
{
    public InputActionReference MovementAction;
    public InputActionReference JumpAction;

    Camera _camera;
    PlayerInput _playerInput;

    public override void ReadInputsFromAsset(out MyInput inputs)
    {
        // Read raw input
        var moveRaw = _playerInput.actions[MovementAction.name].ReadValue<Vector2>();

        // Convert to world-space relative to camera
        Transform cam = _camera.transform;
        float3 worldMove = cam.forward * moveRaw.y + cam.right * moveRaw.x;
        worldMove -= math.projectsafe(worldMove, math.up()); // Remove vertical component
        inputs.Movement = math.normalizesafe(worldMove);

        // Read trigger
        inputs.JumpTrigger = _playerInput.actions[JumpAction.name].WasPerformedThisFrame() ? 1f : 0f;
    }

    public override void InitInputAsset(PlayerGameObjectInstance instance)
    {
        var playerGo = instance.Instance.Value;
        _camera = playerGo.GetComponent<PlayerGoAuthoring>().Camera;
        _playerInput = playerGo.GetComponent<PlayerInput>();
        _playerInput.ActivateInput();
    }
}

    Key points

    • ReadInputsFromAsset is called every Update frame (not FixedUpdate). Return a fresh TInput each call.
    • InitInputAsset is called once when the player is instantiated. Use it to cache references to the camera, PlayerInput, and any other managed objects.
    • The asset is instantiated per player at runtime (ScriptableObject.Instantiate), so each player gets their own cached references. Multiple players on the same machine each have their own copy.

    Camera-Relative Input

    Most 3D games convert 2D stick/keyboard input to world-space movement relative to the camera. The pattern is:

    1. Read the 2D input vector
    2. Multiply by the camera's forward and right vectors
    3. Project onto the horizontal plane (remove the gravity-axis component)
    4. Normalize

    This ensures "push up on the stick" always means "move away from the camera," regardless of camera angle.

    ICharacterLocomotionInput

    Your input struct must implement this interface:

    public struct MyInput : ICharacterLocomotionInput<MyInput>
{
    public float3 Movement;
    public float JumpTrigger;

    public void ApplyInputsToCharacter(in MyInput playerInputs)
    {
        // Continuous inputs: overwrite with latest value
        Movement = playerInputs.Movement;

        // Trigger inputs: ACCUMULATE, don't overwrite
        JumpTrigger += playerInputs.JumpTrigger;
    }
}

    Input Accumulation

    The critical distinction is between continuous and trigger inputs:

    Type Example Behavior Why
    Continuous Movement direction, aim angle Overwrite each frame Only the latest value matters
    Trigger Jump press, fire button Accumulate across frames A press during a non-FixedUpdate frame must not be lost

    If you overwrite JumpTrigger instead of accumulating it, and the player presses Jump during a frame where no FixedUpdate runs, the press is silently discarded.

    Consuming Triggers

    In your state's OnIntegrateState, after reading the trigger value, reset it to zero:

    internal static void OnIntegrateState(ref MyContext ctx)
{
    if (ctx.Input.JumpTrigger > 0)
    {
        // Apply jump
        Jump(ref ctx);
    }
    // Consume the trigger
    MyInput input = ctx.Input;
    input.JumpTrigger = 0;
    ctx.Input = input;
}

    Generated Input Reader System

    The source generator creates an input reader system that:

    1. Runs each Update frame (not FixedUpdate)
    2. For each player, calls PlayerInputAsset.ReadInputsFromAsset to get raw input
    3. For each character controlled by that player, calls ICharacterLocomotionInput.ApplyInputsToCharacter to write to the ECS component

    You never need to write or modify this system. It is produced automatically from your [LocomotionStateMachine] annotation.

    AI and Network Input

    Because input is just an ECS component, you can drive characters from any source:

    AI Input

    Create a system that writes directly to the TInput component without going through PlayerInputAsset:

    [BurstCompile]
partial struct AIInputSystem : ISystem
{
    [BurstCompile]
    public void OnUpdate(ref SystemState state)
    {
        foreach (var (input, transform) in
            SystemAPI.Query<RefRW<MyInput>, RefRO<LocalTransform>>()
                .WithAll<AIControlled>())
        {
            // AI decision-making logic
            input.ValueRW.Movement = ComputeAIMovement(transform.ValueRO);
            input.ValueRW.JumpTrigger = ShouldJump() ? 1f : 0f;
        }
    }
}

    Network Input

    For networked games, the client sends input to the server, which writes it to the TInput component. The locomotion job runs identically on both client and server, producing deterministic results (assuming the same input and physics state).

    Cinemachine Integration

    The sample's LocomotionPlayerInputAsset demonstrates Cinemachine camera setup:

    1. Find all CinemachineVirtualCameraBase components on the player prefab
    2. Set their Follow and LookAt targets to the character's camera target transform
    3. Configure the CinemachineBrain to follow the player's virtual cameras
    4. Assign CinemachineInputAxisController player indices for per-player camera control

    This ensures each player in a split-screen setup has their own camera pipeline.
