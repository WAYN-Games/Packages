    最佳实践

    使用 MGM Character Controller 构建时的固执建议。

    项目结构

    每种角色原型一个状态机

    为具有根本不同移动方式的角色创建单独的状态机类型（例如，地面角色 vs 载具 vs 飞行）。不要试图将所有移动塞进一个状态枚举中。

    保持状态小而专注

    每个状态应处理一种移动行为。如果一个状态有很多分支（"如果在地面上做 X，否则如果在水中做 Y，否则..."），将其拆分为具有转换的独立状态。

    为每个状态使用子 profile

    按使用参数的状态分组 profile 参数：

    public partial struct MyProfile : ILocomotionProfile
{
    public MovingProfile Moving;
    public FallingProfile Falling;
    public JumpingProfile Jumping;
    public DashProfile Dash;
}

    这使 Inspector 保持整洁，并清楚地表明哪些参数影响哪个状态。

    状态逻辑

    始终使用 ref 访问 BaseContext

    在状态方法顶部声明 ref 局部变量以避免复制-修改-写回的开销：

    public void OnIntegrateState(ref MyContext ctx)
{
    ref BaseLocomotionContext baseContext = ref ctx.BaseContext;
    // All modifications go through the ref
}

    例外：OnResolveStateCollisions 需要复制-修改-写回，因为 Move() 在内部修改 context。

    累积触发器，覆盖持续输入

    在 ApplyInputsToCharacter 中：

    • 持续（移动方向、瞄准）：用最新值覆盖
    • 触发（跳跃、冲刺、开火）：用 += 累积

    始终在状态逻辑中读取触发器后将其重置为零来消费。

    共享计算逻辑，不共享状态

    使用 internal static 辅助方法跨状态共享物理计算（例如 FallingState.ComputeGravityAcceleration）。不要共享状态数据——每个状态应通过 OnEnterState / OnExitState 管理自己的初始化和清理。

    不要手动添加平台速度

    ComputeInheritedDisplacement 步骤自动处理平台速度。在 OnIntegrateState 中添加会导致双重应用。唯一的例外是空中状态的 OnEnterState（例如，从平台上跳下），在这里你将 PlatformHorizontalVelocity 添加到角色自身的速度以保持动量。

    转换

    使用 [LocomotionTransition] 属性模式

    将转换声明为带 [LocomotionTransition] 的小结构体，并使用 [EvaluateBefore] / [EvaluateAfter] 控制优先级：

    [LocomotionTransition]
[EvaluateBefore(typeof(IdleToMoving))]
public struct IdleToJumping : ILocomotionTransition<IdleState, JumpingState>
{
    public bool ShouldTransition(ref MyContext ctx)
        => ctx.BaseContext.IsGroundingContact && ctx.Input.JumpTrigger > 0;
}

    这比内联的 EvaluateTransitions 方法更清晰，并让源代码生成器处理分发。

    将关键转换放在前面

    排序转换使最重要的先评估。典型优先级：跳跃 > 移动 > 下落 > 滑行。在跳跃上对移动使用 [EvaluateBefore] 确保在奔跑时按跳跃有效。

    性能

    保持所有状态代码 Burst 兼容

    • 避免在状态逻辑中使用托管类型（字符串、类、数组）——状态在 Burst 编译的作业内运行
    • 状态、context、profile、输入或状态数据结构体中不能有托管类型（string、class、数组）
    • 热路径上不使用 Debug.Log——使用 [Conditional("ENABLE_UNITY_COLLECTIONS_CHECKS")] 保护

    使用共享碰撞体

    默认情况下，胶囊碰撞体在具有相同几何体和过滤器设置的实体之间共享。只有在需要每个实体运行时形状修改时才添加 ForceUniqueColliderAuthoring。

    使用共享的 profile blob 资产

    所有引用相同 profile 资产的实体共享同一块 blob 内存。设计你的 profile 使得常见角色类型使用相同的资产。

    只在调查时启用调试可视化

    调试组件添加动态缓冲区和每帧记录开销。从你不在积极调试的实体中移除 LocomotionDebugConfigAuthoring。

    调试

    在添加日志之前使用调试可视化

    内置的调试可视化按执行阶段显示接触点、移动路径和冲量。这比日志记录更快，并且在 Burst 作业内工作。

    当状态行为异常时检查转换优先级

    如果角色进入了错误的状态，检查 [EvaluateBefore] 排序。优先级更高的转换可能在你期望的转换之前触发了。

    验证接触类型

    许多问题来自不正确的接触分类。记录或检查 ctx.BaseContext.LocomotionContact.ValueRO.ContactType 以确认表面是 Ground、Wall 还是 Ceiling。

    团队工作流

    每种角色类别一个 profile 资产

    使用 ScriptableObject profile 资产，以便设计师可以在不接触代码的情况下调整参数。将这些资产纳入版本控制。

    描述性地命名状态和转换

    对转换使用 {Source}To{Target} 命名约定（例如 IdleToJumping、FallingToMoving）。这使转换图一目了然。

    隔离测试状态

    添加新状态时，先只设置一个进入转换和一个退出转换。验证核心行为有效后再连接所有转换。
