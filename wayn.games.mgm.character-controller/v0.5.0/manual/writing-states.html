    Writing Custom States

    States are the building blocks of character behavior. Each state defines how the character moves, reacts to physics, and transitions to other states. This guide covers the state interface, lifecycle methods, and common patterns.

    The State Interface

    Every state is an unmanaged struct that implements ILocomotionState:

    [LocomotionState(typeof(MyCharacterStateMachine))]
internal partial struct MyState
{
    public void OnEnterState(ref MyContext ctx) { }

    public void OnIntegrateState(ref MyContext ctx) { }

    public void OnResolveStateCollisions(ref MyContext ctx) { }

    public void OnExitState(ref MyContext ctx) { }
}

    Key rules:

    • All lifecycle methods are instance methods with default interface implementations (DIM pattern). They are public void, not internal static. The ILocomotionState interface is added by the source generator via the partial mechanism.
    • Transitions are handled by separate ILocomotionTransition structs (see Transitions below) - the state struct does not define transition logic
    • The source generator dispatches calls to the correct state struct - no virtual dispatch at runtime
    • State methods run inside an already Burst-compiled job, so explicit [BurstCompile] on state structs and methods is not required
    • No managed types (strings, classes, arrays) - this code runs inside Burst jobs

    Lifecycle Methods

    OnEnterState

    Called once when the state machine transitions into this state. Use it to:

    • Initialize state-specific data (reset timers, counters)
    • Set initial velocities
    • Clear contact information (for airborne states)
    public void OnEnterState(ref MyContext ctx)
{
    // Entering a grounded state: cancel all vertical momentum
    ctx.BaseContext.VerticalVelocity = float3.zero;
    ctx.BaseContext.VerticalDisplacement = float3.zero;
}

    OnIntegrateState

    Called once per frame during the integration phase. This is where you compute the character's desired displacement based on input, gravity, and profile parameters.

    Your job here is to populate HorizontalDisplacement, VerticalDisplacement, HorizontalVelocity, and VerticalVelocity on the BaseLocomotionContext.

    The easiest way is to compute accelerations and call UpdateVelocitiesAndComputeDisplacements:

    public void OnIntegrateState(ref MyContext ctx)
{
    ref BaseLocomotionContext baseContext = ref ctx.BaseContext;

    // Compute gravity
    float3 gravityAcceleration = baseContext.GravityDirection * baseContext.GravityStrength;

    // Compute movement acceleration from input
    float3 desiredVelocity = ctx.Input.MovementDirection * ctx.Profile.MaxSpeed;
    float accel = ctx.Profile.MaxSpeed / ctx.Profile.TimeToMaxSpeed;
    MathematicsHelpers.ComputeAccelerationToTargetVelocity(
        out float3 moveAcceleration,
        baseContext.HorizontalVelocity,
        desiredVelocity,
        baseContext.TimeStep,
        accel);

    // Apply both accelerations
    baseContext.UpdateVelocitiesAndComputeDisplacements(
        newHorizontalAcceleration: moveAcceleration,
        newVerticalAcceleration: gravityAcceleration);
}
    Important

    Do not add platform velocity here. The ComputeInheritedDisplacement step handles it automatically after collision resolution. Adding it manually would cause double-application.

    The Context Copy Pattern

    Because BaseLocomotionContext is a value type on the context interface, modifying it requires either:

    Option A: Use a ref local (when your context exposes the backing field):

    ref BaseLocomotionContext baseContext = ref ctx.BaseContext;
baseContext.HorizontalVelocity = float3.zero;
// Changes are written through the ref

    Option B: Copy, modify, write back (when going through the interface):

    BaseLocomotionContext baseContext = ctx.BaseContext;
baseContext.HorizontalVelocity = float3.zero;
ctx.BaseContext = baseContext;

    The sample uses ref access because the context struct exposes BaseContext as a public field in addition to the interface property.

    OnResolveStateCollisions

    Called potentially multiple times per frame (up to MaxBounce iterations). Each call represents one iteration of the collide-and-slide algorithm.

    In most cases, your implementation simply delegates to baseContext.Move():

    public void OnResolveStateCollisions(ref MyContext ctx)
{
    BaseLocomotionContext baseContext = ctx.BaseContext;
    baseContext.Move();
    ctx.BaseContext = baseContext;
}
    Note

    You must use the copy-modify-writeback pattern here because Move() modifies the context's position, displacement, and contact data internally.

    For advanced use cases, you can add custom collision response logic. For example, applying an impulse to dynamic bodies the character hits:

    public void OnResolveStateCollisions(ref MyContext ctx)
{
    BaseLocomotionContext baseContext = ctx.BaseContext;
    baseContext.Move();

    // Push dynamic bodies away on contact
    if (baseContext.IsInContact &&
        baseContext.LocomotionContact.ValueRO.BodyType == BodyType.Dynamic)
    {
        float3 pushVelocity = baseContext.HorizontalVelocity * 0.5f;
        baseContext.ApplyImpulse(
            ref baseContext,
            baseContext.LocomotionContact.ValueRO.BodyIndex,
            baseContext.LocomotionContact.ValueRO.Entity,
            /* impactorBodyIndex */ ...,
            baseContext.CharacterEntity,
            pushVelocity);
    }

    ctx.BaseContext = baseContext;
}

    OnExitState

    Called once when the state machine transitions out of this state. Use it to clean up state-specific data:

    public void OnExitState(ref MyContext ctx)
{
    // Reset jump counter when leaving the jumping state
    ctx.StateData.JumpData.JumpsCount = 0;
}

    Transitions

    Transitions define when and how the character moves between states. The package supports two patterns.

    Recommended: Attribute-Based Transitions

    Declare transitions as small structs with [LocomotionTransition]. The source generator discovers them and handles dispatch, OnExitState, and OnEnterState automatically.

    [LocomotionTransition]
[EvaluateBefore(typeof(IdleToMoving))]
[EvaluateBefore(typeof(IdleToFalling))]
public struct IdleToJumping : ILocomotionTransition<IdleState, JumpingState>
{
    public bool ShouldTransition(ref MyContext ctx)
        => ctx.BaseContext.IsGroundingContact && ctx.Input.JumpTrigger > 0;
}

    Key rules:

    1. The struct name follows the {Source}To{Target} convention
    2. ILocomotionTransition<TFrom, TTo> declares which states are connected
    3. ShouldTransition returns true when the transition should fire
    4. [EvaluateBefore] / [EvaluateAfter] control priority - the first matching transition wins
    5. The source generator calls OnExitState on the source and OnEnterState on the target automatically

    See the sample's LocomotionTransitions.cs for a complete example with 17 transitions.

    The attribute-based pattern is the only supported approach. The source generator discovers all [LocomotionTransition] structs, handles exit/enter calls, and makes transition priority explicit via [EvaluateBefore]/[EvaluateAfter] attributes.

    Consuming Trigger Input

    Trigger inputs like jump presses must be consumed after use. Otherwise, the trigger fires every frame:

    static void ConsumeJumpTrigger(ref MyContext ctx)
{
    MyInput value = ctx.Input;
    value.JumpTrigger = 0;
    ctx.Input = value;
}

    Call this in OnIntegrateState after using the trigger value. The copy-modify-writeback pattern is required because Input is a value type on the context.

    Reusing Logic Across States

    States can share logic via internal static helper methods. For example, the JumpingState in the sample reuses FallingState.ComputeGravityAcceleration for gravity and MovingState.ComputeHorizontalMovementAcceleration for mid-air movement:

    public void OnIntegrateState(ref MyContext ctx)
{
    // Reuse falling gravity computation
    FallingState.ComputeGravityAcceleration(ref ctx, terminalVelocity, out float3 gravityAccel);

    // Reuse moving horizontal acceleration
    MovingState.ComputeHorizontalMovementAcceleration(
        ref ctx.BaseContext, ctx.Profile.Moving,
        ctx.Input.MovementDirection, ctx.Input.MovementMagnitude,
        out float3 moveAccel);

    ctx.BaseContext.UpdateVelocitiesAndComputeDisplacements(moveAccel, gravityAccel);
}

    This avoids duplicating physics logic while keeping each state self-contained.

    Common State Patterns

    Grounded State (Idle / Moving)

    • OnEnterState: Zero out vertical velocity
    • OnIntegrateState: Compute horizontal acceleration only
    • Transitions: Check for jump input, loss of ground contact, slope above max

    Airborne State (Falling / Jumping)

    • OnEnterState: Clear contact (LocomotionContact.None), optionally inherit platform velocity
    • OnIntegrateState: Apply gravity + optional air control
    • Transitions: Check for ground contact, wall contact

    Sliding State

    • OnIntegrateState: Apply gravity (character slides down slope)
    • Transitions: Check for loss of contact (to falling), ground contact below max slope (to moving/idle)

    Custom State Ideas

    • Wall Running: Enter when jumping into a wall at speed; apply gravity slowly; allow wall jump
    • Dashing: Enter on trigger; apply burst horizontal velocity; exit after timer
    • Swimming: Enter when submerged; replace gravity with buoyancy; allow 3D movement
    • Climbing: Enter at ladder; disable gravity; move along ladder axis

    Debugging States

    If your state isn't behaving as expected:

    1. Check transition priority order - an earlier transition might be firing first
    2. Log baseContext.LocomotionContact.ValueRO.ContactType to verify contact detection
    3. Verify that OnExitState and OnEnterState are called during transitions
    4. Ensure trigger inputs are accumulated in ApplyInputsToCharacter and consumed in OnIntegrateState
    5. Use the debug visualization to see contact points, movement paths, and impulses
