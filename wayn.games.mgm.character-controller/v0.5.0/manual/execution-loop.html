    The Execution Loop

    Every fixed update, the locomotion job processes each character entity through a precise sequence of steps. Understanding this sequence is critical for writing correct state logic.

    Overview

    The locomotion job runs as a Burst-compiled IJobChunk scheduled in the FixedStepSimulationSystemGroup, after the physics step. Each character is processed independently and in parallel.

    The steps execute in this exact order:

    flowchart TD
    START(["Frame Start"])

    subgraph S1["1. HandleExternalImpulses"]
        HEI["Accumulate velocity impulses<br/>from CharacterContactSystem"]
    end

    subgraph S2["2. Integrate - <code>OnIntegrateState</code>"]
        INT["Compute velocities + displacement"]
    end

    subgraph S3["3. SolveOverlaps"]
        SO["Multi-plane de-collision<br/>Sets OverlapState: None | Push | Pinch | Crush"]
    end

    subgraph S4["4. CollideAndSlide - <code>OnResolveStateCollisions</code>"]
        CAS_CRUSH{"OverlapState<br/>== Crush?"}
        CAS_LOOP{"HasRemaining-<br/>Displacement &amp;&amp;<br/>bounce &lt; MaxBounce?"}
        CAS["Project, cast, slide,<br/>update contacts"]
        CAS_CRUSH -- Yes --> CAS_END
        CAS_CRUSH -- No --> CAS_LOOP
        CAS_LOOP -- Yes --> CAS --> CAS_LOOP
        CAS_LOOP -- No --> CAS_END
        CAS_END([done])
    end

    subgraph S5["5. SnapToGround"]
        SNAP_NONE{"ContactType<br/>== None?"}
        SNAP_UP{"Moving upward?<br/><i>dot(vVel, gravity) &lt; −ε</i>"}
        SNAP_VVEL{"VerticalVelocity<br/>== 0?"}
        SNAP["Capsule cast downward<br/>Snap position + update contact"]
        SNAP_NONE -- "Yes (skip)" --> SNAP_END
        SNAP_NONE -- No --> SNAP_UP
        SNAP_UP -- "Yes (clear contact)" --> SNAP_END
        SNAP_UP -- No --> SNAP_VVEL
        SNAP_VVEL -- "No (skip)" --> SNAP_END
        SNAP_VVEL -- "Yes (at rest)" --> SNAP --> SNAP_END
        SNAP_END([done])
    end

    subgraph S6["6. InheritDisplacementFromMotionVelocity"]
        IH_GROUND{"Grounding<br/>contact?"}
        IH_CRUSH{"Crush?"}
        IH_DYN{"Standing on<br/>dynamic body?"}
        IH["Compute + apply platform displacement<br/>Apply twist rotation"]
        IH_GROUND -- No --> IH_END
        IH_GROUND -- Yes --> IH_CRUSH
        IH_CRUSH -- Yes --> IH_END
        IH_CRUSH -- No --> IH_DYN
        IH_DYN -- No --> IH_END
        IH_DYN -- Yes --> IH --> IH_END
        IH_END([done])
    end

    subgraph S7["7. EvaluateTransition"]
        ET["Source-generated dispatch<br/>to LocomotionTransition structs"]
    end

    DONE(["Frame End"])

    START --> HEI
    HEI --> INT
    INT --> SO
    SO --> CAS_CRUSH
    CAS_END --> SNAP_NONE
    SNAP_END --> IH_GROUND
    IH_END --> ET
    ET --> DONE

    Each step in the pipeline is a default interface method (DIM) on ILocomotionStateMachine that can be overridden on your state machine's partial struct. See Customizing the Execution Pipeline for details.

    There is no OnUpdateState method. The integration and collision resolution phases together form the "update" of your state. Transitions are evaluated once, at the very end, after all physics is resolved.

    Step-by-Step Breakdown

    1. HandleExternalImpulses

    The CharacterContactSystem runs during the PhysicsSimulationGroup (between contact creation and jacobian creation) and collects velocity impulses from external sources, such as:

    • Dynamic rigid bodies that have collided with the character
    • Other characters pushing against this one
    • Explosions or force fields

    These impulses are applied to the character's velocity before your state logic runs, so your OnIntegrateState sees the updated velocity. This is why you should not clear velocity unconditionally at the start of integration unless you intend to discard external forces.

    2. Integrate (OnIntegrateState)

    This is where your state logic computes velocities and displacements for the current frame. The system calls your state's OnIntegrateState method.

    In this step, you should:

    • Compute acceleration from input (movement direction, jump force)
    • Apply gravity acceleration
    • Call baseContext.UpdateVelocitiesAndComputeDisplacements() to convert velocities into displacements

    What you should NOT do here:

    • Move the character's position directly (that happens in the collision step)
    • Add platform velocity (the ComputeInheritedDisplacement step handles that automatically)
    • Read contact information from this frame (contacts haven't been resolved yet; you see last frame's contacts)

    The Kinematic Equation

    UpdateVelocitiesAndComputeDisplacements applies the standard kinematic equations:

    displacement += velocity * dt + 0.5f * acceleration * dt * dt;
velocity += acceleration * dt;

    This is done separately for horizontal and vertical components, allowing different behavior for each (e.g., gravity only affects vertical, movement input only affects horizontal).

    3. SolveOverlaps

    After integration, the system checks whether the character's collider is currently overlapping with any other collider. This can happen when:

    • A moving platform pushes the character into a wall
    • Another character occupies the same space
    • The character was teleported into geometry

    The overlap solver attempts to push the character out of the overlap. If it cannot (e.g., the character is pinched between two moving surfaces), it tracks the overlap state:

    OverlapState Meaning
    None No overlap detected
    Push Character was pushed out successfully
    Pinch Character is squeezed but not fully trapped
    Crush Character is fully embedded with no escape

    You can read ctx.BaseContext.OverlapState in your state logic to react to crushing situations (e.g., kill the character).

    4. CollideAndSlide (OnResolveStateCollisions)

    After integration computes the desired displacement, the character needs to actually move through the world while respecting colliders. This is the collide-and-slide loop.

    The system invokes your state's OnResolveStateCollisions as long as there is remaining displacement, up to CharacterPhysicsProfile.MaxBounce iterations. This multi-bounce approach allows the character to slide along walls, step over obstacles, and navigate complex geometry.

    Default Behavior

    If you do not override OnResolveStateCollisions, the default DIM calls baseContext.Move(), which performs two operations:

    1. Displacement correction - Adjusts HorizontalDisplacement and VerticalDisplacement based on the current contact type:

      Contact Type Behavior
      None Displacement is applied as-is (free movement in air)
      Ground Vertical displacement is cancelled; horizontal displacement follows the ground slope
      Wall Step detection is attempted; if not a step, displacement is projected along the wall surface
      Ceiling Vertical displacement is projected along the ceiling; horizontal displacement is constrained

    2. Cast and move - Calls PhysicsOperations.MoveUpToNextObstacle to cast the capsule forward by the corrected displacement, move the character to the hit point (or the full distance if nothing was hit), and update contact information.

    To customize how collisions are resolved, see Custom Collision Response.

    Step Detection

    When a wall contact is detected, the default Move() checks whether it could be a step:

    1. The character must be in contact with a surface
    2. The slope must exceed MaxSlope
    3. The character must be moving into the slope
    4. The contact height must be below MaxStepHeight

    If all conditions are met, PhysicsOperations.TryStep attempts to move the character up and over the step.

    5. SnapToGround

    After all collision resolution is complete, the system performs a downward capsule cast to detect the ground surface below the character. This ensures the character stays grounded when walking down slopes or stairs, rather than becoming airborne for a frame.

    The snap distance is computed by your context's ComputeGroundSnappingDistance method. The default implementation returns a hardcoded distance of 1f. Override this method on your context's partial struct to account for step height, slope angle, speed, or platform velocity as needed.

    If a ground surface is found within the snap distance, the character's position is adjusted downward and the contact information is updated.

    6. InheritDisplacementFromMotionVelocity

    If the character is standing on a dynamic body (kinematic or fully-simulated), this step computes the displacement and rotation the character inherits from that body's motion velocity.

    The displacement is applied to the character's position in the same frame. The inherited velocities are stored in:

    • baseContext.PlatformHorizontalVelocity
    • baseContext.PlatformVerticalVelocity

    These are available immediately - for example, a jumping state's OnEnterState (called during EvaluateTransition in the same frame) can read PlatformHorizontalVelocity to preserve momentum when jumping off a platform. The position adjustment from platform movement is applied automatically, which is why you should never add platform velocity manually in OnIntegrateState.

    7. EvaluateTransition

    The final step evaluates all [LocomotionTransition] structs registered for the current state. By this point:

    • The character has moved to its final position
    • All contacts are resolved and up-to-date
    • Platform inheritance is computed

    The source generator dispatches to each transition's ShouldTransition method in the order determined by [EvaluateBefore]/[EvaluateAfter] attributes. The first matching transition wins. The generator automatically handles calling OnExitState on the current state and OnEnterState on the new state.
