    Extension du package

    Le MGM Character Controller est concu pour etre etendu sans modifier le code source du package. Cette page couvre les points d'extension courants et les patrons de conception.

    Points d'extension

    1. Etats personnalises

    L'extension la plus courante. Ajoutez de nouveaux comportements de mouvement (course murale, nage, escalade, dash) en creant de nouvelles structs d'etat et en les ajoutant a votre enum d'etats.

    Voir Ecrire des etats personnalises pour le guide complet.

    2. Profils personnalises

    Votre struct ILocomotionProfile peut contenir n'importe quelle donnee non geree. Ajoutez des champs pour de nouveaux parametres de gameplay :

    [Serializable]
public partial struct MyProfile : ILocomotionProfile
{
    public MovingProfile Moving;
    public FallingProfile Falling;
    public JumpingProfile Jumping;

    // Custom extension
    public SwimmingProfile Swimming;
    public DashProfile Dash;
    public WallRunProfile WallRun;
}

    Chaque sous-profil est une struct simple avec des champs de configuration. Les donnees sont cuites dans un blob asset, ajoutez donc les champs correspondants a votre LocomotionProfileAsset et a la methode BuildBlobAsset.

    3. Contexte personnalise

    Votre struct de contexte peut inclure des proprietes calculees supplementaires et des methodes d'aide :

    public partial struct MyContext : ILocomotionContext<MyProfile, MyInput, MyStateData>
{
    public BaseLocomotionContext BaseContext;
    public MyStateData StateData;

    // Standard interface properties...

    // Custom helpers
    public bool IsUnderwater => BaseContext.CharacterPosition.Position.y < WaterLevel;
    public bool WantsToMove => Input.MovementMagnitude > math.EPSILON;
    public bool CanDash => StateData.DashCooldown <= 0;

    // Custom ground snap for swimming (no ground snap when submerged)
    public void ComputeGroundSnappingDistance(in MyProfile profile, out float distance)
    {
        if (IsUnderwater)
        {
            distance = 0; // No ground snapping while swimming
            return;
        }
        // Normal ground snap calculation...
    }
}

    4. StateData personnalise

    Ajoutez des donnees persistantes par etat pour tout suivi specifique a un etat :

    public partial struct MyStateData : IComponentData
{
    public JumpData JumpData;
    public float DashCooldown;
    public float WallRunTimer;
    public float3 WallNormal;  // Remembered from wall run entry
}

    Les donnees d'etat persistent entre les frames et les transitions d'etat. Nettoyez-les dans OnEnterState ou OnExitState selon les besoins.

    5. Input personnalise

    Ajoutez des canaux d'entree pour de nouvelles actions :

    public partial struct MyInput : ICharacterLocomotionInput<MyInput>
{
    public float3 Movement;
    public float JumpTrigger;
    public float DashTrigger;      // New action
    public float InteractTrigger;  // New action

    public void ApplyInputsToCharacter(in MyInput playerInputs)
    {
        Movement = playerInputs.Movement;
        JumpTrigger += playerInputs.JumpTrigger;       // Accumulate triggers
        DashTrigger += playerInputs.DashTrigger;
        InteractTrigger += playerInputs.InteractTrigger;
    }
}

    N'oubliez pas de lire les nouveaux inputs dans votre sous-classe PlayerInputAsset et de consommer les declencheurs dans votre logique d'etat.

    6. Accrochage au sol personnalise

    Surchargez ComputeGroundSnappingDistance sur votre contexte pour controler quand et a quelle distance le personnage s'accroche au sol :

    public void ComputeGroundSnappingDistance(in MyProfile profile, out float distance)
{
    // Disable ground snapping while jumping upward
    if (StateData.CurrentState == MyStates.Jumping &&
        math.dot(BaseContext.VerticalVelocity, -BaseContext.GravityDirection) > 0)
    {
        distance = 0;
        return;
    }

    // Default calculation
    var slopeDistance = math.length(BaseContext.HorizontalVelocity * BaseContext.TimeStep) *
                       math.tan(math.radians(CharacterPhysicsProfile.MaxSlope));
    distance = math.max(CharacterPhysicsProfile.MaxStepHeight, slopeDistance);
}

    7. Reponse de collision personnalisee

    Surchargez OnResolveStateCollisions pour ajouter un comportement personnalise lorsque le personnage entre en contact avec quelque chose :

    public void OnResolveStateCollisions(ref MyContext ctx)
{
    BaseLocomotionContext baseContext = ctx.BaseContext;
    baseContext.Move();

    // Custom: bounce off trampolines
    if (baseContext.IsGroundingContact &&
        HasTrampolineTag(baseContext.LocomotionContact.ValueRO.Entity))
    {
        baseContext.VerticalVelocity = -baseContext.VerticalVelocity * 1.5f;
    }

    ctx.BaseContext = baseContext;
}

    8. Systemes personnalises

    Vous pouvez ajouter des systemes qui modifient les composants de locomotion a n'importe quel point du pipeline :

    [UpdateInGroup(typeof(FixedStepSimulationSystemGroup))]
[UpdateBefore(typeof(MyLocomotionSystem))]
partial struct GravityZoneSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        foreach (var gravity in SystemAPI.Query<RefRW<LocomotionGravity>>()
            .WithAll<InGravityZone>())
        {
            gravity.ValueRW.Gravity = new float3(0, -4.9f, 0); // Half gravity
        }
    }
}

    Personnalisation du pipeline d'execution

    Chaque etape de la boucle d'execution est une methode d'interface par defaut (DIM) sur ILocomotionStateMachine. Vous pouvez surcharger n'importe laquelle d'entre elles sur la struct partielle de votre machine a etats pour remplacer ou encapsuler le comportement par defaut.

    Le generateur de source produit une struct partielle pour votre machine a etats (par exemple, MyStateMachine). Comme elle est partielle, vous pouvez ajouter une seconde declaration partielle qui surcharge n'importe quelle DIM :

    public partial struct MyStateMachine
{
    public void Integrate(ref MyStateMachine stateMachine, ref MyContext context)
    {
        // Skip movement entirely when crushed
        if (context.BaseContext.OverlapState == OverlapState.Crush)
            return;

        // Otherwise run the default per-state integration
        stateMachine.IntegratePerState(ref context);
    }
}

    Methodes de pipeline surchargeable

    Methode Comportement par defaut Surcharger pour...
    Execute Orchestre le pipeline complet (toutes les etapes ci-dessous dans l'ordre) Remplacer la boucle d'execution entiere
    HandleExternalImpulses Accumule les impulsions de velocite des contacts de corps Filtrage ou mise a l'echelle personnalises des impulsions
    Integrate Appelle IntegratePerState (saute si ecrase) Ajouter une logique pre/post integration
    SolveOverlaps Algorithme de de-collision multi-plan Resolution de chevauchement personnalisee
    CollideAndSlide Boucle ResolveCollisionPerState jusqu'a MaxBounce Iteration de collision personnalisee
    SnapToGround Capsule cast descendant pour maintenir le contact au sol Adhesion au sol personnalisee
    InheritDisplacementFromMotionVelocity Calcule l'heritage de velocite de la plateforme a partir des corps physiques Comportement personnalise sur les plateformes
    EvaluateTransition Appelle EvaluateTransitionPerState Ajouter une logique pre/post transition
    Note

    IntegratePerState, ResolveCollisionPerState et EvaluateTransitionPerState sont generes par le generateur de source (dispatch via switch vers les methodes par etat). Ils ne sont pas destines a etre surcharges directement.

    Voir La boucle d'execution pour l'ordre et le flux de donnees entre ces etapes.

    Personnalisation du systeme de locomotion

    Le generateur de source cree une interface partielle I{Prefix}LocomotionSystem qui herite de ILocomotionSystem<...>. La struct de systeme generee implemente cette interface avec des methodes de cycle de vie DIM que vous pouvez surcharger.

    Pour personnaliser, ajoutez votre propre declaration d'interface partielle :

    public partial interface IMyLocomotionSystem
{
    void ILocomotionSystem<MyStateMachine, MyStates, MyContext, MyProfile, MyInput, MyStateData>
        .OnLocomotionSystemUpdate(
            ref LocomotionJobData jobData, ref EntityQuery query, ref SystemState state)
    {
        if (query.IsEmptyIgnoreFilter) return;
        jobData = jobData.Update(ref state);

        // Schedule single-threaded instead of parallel (useful for debugging)
        state.Dependency = ScheduleLocomotionJob(
            state.Dependency,
            state.WorldUnmanaged.Time.DeltaTime,
            jobData, query,
            jobData.GetPhysicsWorldSingleton());
    }
}

    Methodes de cycle de vie du systeme surchargeable

    Methode Comportement par defaut Surcharger pour...
    OnLocomotionSystemCreate Construit la requete d'entite et cree LocomotionJobData Ajouter RequireForUpdate, des filtres de requete personnalises, la configuration du rate manager
    OnLocomotionSystemUpdate Met a jour les donnees du job et appelle ScheduleParallelLocomotionJob Utiliser un scheduling mono-thread, ajouter un saut conditionnel, integrer un rate manager
    OnLocomotionSystemDestroy No-op Nettoyage personnalise des ressources
    Note

    Les implementations DIM par defaut utilisent des parametres ref au lieu de SystemAPI car le generateur de source de Unity ne peut pas patcher les sites d'appel dans le code emis par le generateur de source du package (les deux s'executent en parallele). Vos propres surcharges d'interface partielle peuvent utiliser SystemAPI puisqu'elles sont visibles par le generateur de Unity.

    Machines a etats multiples

    Vous pouvez avoir plusieurs types de machines a etats independants dans le meme projet. Par exemple :

    • GroundLocomotionStates - Marche, course, saut, chute
    • VehicleStates - Conduite, freinage, boost, vol
    • SwimmingStates - Surface, submerge, plongee

    Chacun a son propre enum, contexte, profil, input, donnees d'etat et systemes generes. Ils sont completement independants.

    Pour passer un personnage d'une machine a etats a une autre, vous devez ajouter/supprimer les composants pertinents (le composant de machine a etats, l'input, le profil, les donnees d'etat). C'est un patron avance qui implique des changements structurels d'entite.

    Travailler avec le generateur de source

    Le generateur de source Roslyn lit les attributs sur votre struct d'ancrage et vos structs d'etat et genere :

    1. Le composant de machine a etats avec la logique de dispatch
    2. Le systeme de lecture d'input
    3. Les methodes de cycle de vie ISystem (OnCreate, OnUpdate, OnDestroy) sur votre systeme de locomotion

    Le source du generateur se trouve dans SourceGenerator/SrcGenProj~/. Pour modifier les patrons de code genere :

    1. Editez StateMachineSourceGenerator.cs
    2. Compilez avec dotnet build depuis le repertoire SrcGenProj~
    3. La DLL est copiee dans SourceGenerator/ et prise en charge par Unity
    Warning

    Modifier le generateur de source affecte toutes les machines a etats du projet. Testez minutieusement.

    Creation de personnages composites

    Pour les personnages necessitant a la fois une locomotion ECS et des hierarchies visuelles complexes, utilisez le patron CompositeEntityAuthoring :

    1. L'entite Locomotor porte tous les composants physiques et de machine a etats
    2. Une entite Model suit le Locomotor via CharacterFollower
    3. Le GameObject du Model peut avoir des objets enfants avec des Animators, des systemes de particules, des sources audio

    Le CharacterFollowerAuthoring lie le modele au locomotor lors du baking. A l'execution, le CharacterFollowerSystem synchronise la transformation.

    Considerations de performance

    Mise a l'echelle vers de nombreux personnages

    Le job de locomotion s'execute en parallele a travers les chunks. Pour maximiser le debit :

    • Conservez tous les personnages avec le meme archetype (meme ensemble de composants)
    • Utilisez des colliders partages (evitez ForceUniqueColliderAuthoring sauf si necessaire)
    • Profitez des blob assets partages (tous les personnages referencant le meme profil partagent la memoire)
    • N'activez la visualisation de debogage que sur les personnages que vous etudiez activement

    Compilation Burst

    Les methodes d'etat s'executent a l'interieur d'un job compile avec Burst, elles doivent donc etre compatibles Burst. Erreurs courantes empechant la compilation Burst :

    • Utilisation de types string ou List<T>
    • Appel de methodes virtuelles gerees (les methodes de cycle de vie DIM sont dispatchees par le generateur de source, pas via un dispatch virtuel)
    • Acces aux API GameObject ou MonoBehaviour
    • Utilisation de Debug.Log (utilisez des gardes [Conditional("ENABLE_UNITY_COLLECTIONS_CHECKS")])

    Profilage

    Utilisez le Profiler de Unity pour identifier les goulots d'etranglement :

    • Le job de locomotion apparait comme LocomotionJob dans la timeline des Jobs
    • Les marqueurs de profilage par etape sont des champs statiques sur ILocomotionStateMachine, gates par [Conditional("ENABLE_LOCOMOTION_PROFILER")] via ProfilerMarkerExtensions
    • Le LocomotionProfilerHUD affiche le timing au niveau de la frame en jeu
