    Guide de mise a jour

    Cette page documente les changements incompatibles et les etapes de migration entre les versions.

    0.3.0 vers 0.4.0

    Date de publication : 22/03/2026

    Changements incompatibles

    Reecriture du generateur de source

    Le generateur de source a ete reecrit. ILocomotionStateMachineDefinition a ete supprime et remplace par une decouverte basee sur les attributs.

    Avant (0.3.0) :

    public struct MyStateMachineDefinition : ILocomotionStateMachineDefinition { ... }

    Apres (0.4.0) :

    [LocomotionStateMachine]
public partial struct MyStateMachine { }

    Migration : Supprimez votre ancienne struct de definition. Creez une nouvelle struct avec l'attribut [LocomotionStateMachine]. Le generateur de source decouvre les etats et les transitions a partir des attributs [LocomotionState] et [LocomotionTransition].

    Valeurs par defaut du cycle de vie DIM

    Le generateur de source emet desormais des methodes de transfert de struct explicites pour les hooks de cycle de vie des methodes d'interface par defaut. Si vous vous appuyiez sur les valeurs par defaut DIM appelees implicitement, cela devrait fonctionner de la meme maniere. Si vous aviez surcharge la DIM sur une struct partielle, verifiez que vos surcharges compilent toujours.

    Reorganisation du menu de l'editeur

    Les menus de creation, les modeles de script et les utilitaires ont ete deplaces dans le sous-dossier Editor/CreationMenus/. Si vous aviez des scripts referencant les anciens chemins, mettez a jour les references.

    Scaffolding de script de locomotion

    Les nouveaux menu CreateLocomotionScript et LocomotionScriptCreationMenus remplacent l'ancien scaffolding. La structure de dossiers generee et le nommage des fichiers ont change :

    • Tous les fichiers generes sont desormais prefixes avec le nom de la machine a etats pour eviter les collisions entre plusieurs machines a etats
    • Les assembly definitions sont auto-generees avec des namespaces et des references GUID corrects

    Migration : Si vous utilisiez l'ancien outil de scaffolding, les fichiers generes fonctionnent toujours. Les nouveaux fichiers generes par l'outil mis a jour utiliseront la nouvelle convention de nommage.

    Comment verifier la mise a jour

    1. Ouvrez le projet dans Unity 6000.3+
    2. Attendez que la compilation se termine
    3. Verifiez la Console pour les erreurs - corrigez les references de type manquantes
    4. Entrez en mode Play avec une scene existante
    5. Verifiez que le mouvement du personnage fonctionne comme avant

    0.2.0 vers 0.3.0

    Date de publication : 18/03/2026

    Changements incompatibles

    ILocomotionState utilise le patron DIM de methodes d'instance

    Les methodes de cycle de vie des etats sont passees de internal static a des methodes d'instance avec des implementations d'interface par defaut.

    Avant (0.2.0) :

    internal static void OnEnterState(ref MyContext ctx) { }

    Apres (0.3.0) :

    public void OnEnterState(ref MyContext ctx) { }

    Migration : Changez toutes les methodes de cycle de vie des etats de internal static a des methodes d'instance public.

    CharacterPhysicsProfile deplace vers ILocomotionContext

    CharacterPhysicsProfile ne fait plus partie de ILocomotionProfile. C'est desormais une propriete de ILocomotionContext.

    Avant : ctx.Profile.PhysicsProfileValue Apres : ctx.CharacterPhysicsProfile

    Migration : Deplacez l'acces au profil physique de ctx.Profile vers ctx.CharacterPhysicsProfile. Supprimez PhysicsProfileValue de votre struct de profil.

    Renommage des classes d'authoring de base

    Ancien nom Nouveau nom
    BaseCharacterAuthoring BaseCharacterLocomotorAuthoring
    BaseCharacterBaker BaseCharacterLocomotorBaker

    Migration : Cherchez et remplacez les anciens noms dans vos scripts d'authoring.

    Heritage de velocite de plateforme

    Le systeme de plateforme base sur le suivi de chemin a ete remplace par une approche basee sur MotionVelocity via ComputeInheritedDisplacement.

    Migration : Si vous utilisiez les anciens composants PlatformPath, supprimez-les. L'heritage de plateforme fonctionne desormais automatiquement pour tout corps dynamique sur lequel le personnage est au sol. Accedez aux donnees de la plateforme via ctx.BaseContext.PlatformHorizontalVelocity et ctx.BaseContext.PlatformVerticalVelocity.

    Timing de l'evaluation des transitions

    EvaluateTransitions s'execute desormais une seule fois a la fin de la boucle d'execution (apres toute la physique), et non plus apres chaque iteration de rebond.

    Migration : Si vos transitions dependent de donnees de contact en cours de frame, adaptez-les pour utiliser l'etat de contact final post-physique.

    Comment verifier

    1. Compilez le projet - corrigez les types renommes et les signatures de methodes modifiees
    2. Entrez en mode Play et testez tous les etats et transitions
    3. Testez les plateformes mobiles - verifiez que les personnages se deplacent correctement dessus
    4. Activez la visualisation de debogage pour confirmer que la detection de contact fonctionne toujours

    0.1.0 vers 0.2.0

    Date de publication : 05/10/2025

    Changements

    • Ajout du generateur de source pour les systemes de machine a etats
    • Ajout du menu de creation de machine a etats
    • Refactoring des systemes pour supporter les machines a etats definies par l'utilisateur

    Migration : L'API 0.1.0 etait interne/preview. Recreez votre machine a etats en utilisant le nouveau menu de creation (Assets > Create > WAYN Games > Locomotion > StateMachine).
