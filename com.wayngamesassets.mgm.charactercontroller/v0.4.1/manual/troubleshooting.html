    Troubleshooting

    This page lists common issues organized by symptom. For each issue: what you see, why it happens, and how to fix it.

    Character falls through the floor

    Cause: The floor does not have a physics collider that the ECS world can see.

    Fix:

    1. Select the floor GameObject inside the sub-scene
    2. Add a Physics Shape component (not a legacy MeshCollider or BoxCollider)
    3. Verify the sub-scene is loaded (check Hierarchy for the sub-scene icon)

    Verify: Enter Play mode. Open the Physics Debugger (Window > Analysis > Physics Debugger) and confirm the floor collider is visible.

    Prevention: Always use Physics Shape / Physics Body components for ECS colliders. Legacy colliders are ignored by Unity.Physics.

    Character doesn't respond to input

    Cause: Input is not flowing from the hardware to the ECS component.

    Fix - check each layer in order:

    1. Input System package: Confirm com.unity.inputsystem is installed and the Active Input Handling is set to Input System Package (New) or Both in Edit > Project Settings > Player
    2. Player entity: Verify the Player entity has the input asset assigned in the Player Character Authoring component
    3. Player prefab: The Player GO Prefab must have a PlayerInput component with input actions assigned
    4. Character reference: The Player entity must reference the correct Character entity

    Verify: Add a breakpoint or log in your PlayerInputAsset.ReadInputsFromAsset() method to confirm it is called each frame.

    Character jitters on the ground

    Cause: The character's visual position updates discretely each fixed step, causing visible stuttering.

    Fix:

    1. Set InterpolationMode to Interpolated in the Character Physics Profile
    2. If still jittering, increase CollisionTolerance slightly (try 0.01)
    3. Verify the ground collider is static (not a dynamic body rebuilding each frame)

    Verify: The jitter should disappear or become imperceptible at 60+ FPS.

    Character walks through walls

    Cause: The collide-and-slide algorithm does not have enough iterations to resolve the collision.

    Fix:

    1. Increase MaxBounce on the Character Physics Profile (try 8-10, default is 6)
    2. Check that the wall's collision layer is included in the character's CollidesWith mask
    3. If the character is moving very fast, the displacement per frame may exceed the collider size - reduce speed or increase the fixed update rate

    Verify: Enable debug visualization with contact points to see if wall contacts are being detected.

    Character cannot climb steps

    Cause: Step detection conditions are not met.

    Fix:

    1. Increase MaxStepHeight on the Character Physics Profile (must be less than the capsule's flat section height: CapsuleHeight - 2 * CapsuleRadius)
    2. Ensure MaxSlope is high enough that the step's vertical face is classified as a wall (not ground)
    3. Verify the step geometry is clean - sloped mesh artifacts can confuse the step detector

    Verify: Enable debug visualization. Wall contacts should appear at the step face, followed by successful upward movement.

    Character slides off walkable slopes

    Cause: The slope angle exceeds the MaxSlope threshold.

    Fix: Increase MaxSlope on the Character Physics Profile. The default handles most cases at 45-60 degrees.

    Verify: Log or inspect ctx.BaseContext.LocomotionContact.ValueRO.Normal to compute the actual slope angle.

    Jump doesn't fire

    Cause: The jump trigger input is being overwritten instead of accumulated, or it is not being consumed after use.

    Fix:

    1. In your ICharacterLocomotionInput.ApplyInputsToCharacter, accumulate the trigger: JumpTrigger += playerInputs.JumpTrigger (not JumpTrigger = playerInputs.JumpTrigger)
    2. In your PlayerInputAsset.ReadInputsFromAsset, return 1f for the trigger only on the frame the button was pressed (WasPerformedThisFrame()), not while it is held

    Verify: Add a temporary log in ApplyInputsToCharacter to see the accumulated value.

    Jump fires every frame

    Cause: The jump trigger is not being consumed (reset to zero) after use.

    Fix: After reading the trigger value in your state's OnIntegrateState, reset it:

    LocomotionInput value = ctx.Input;
value.JumpTrigger = 0;
ctx.Input = value;

    Verify: The character should jump once per button press, not continuously.

    State transition never fires

    Cause: The transition's conditions are not met, or a higher-priority transition is firing first.

    Fix:

    1. Check the [EvaluateBefore] attributes on your transitions - they control evaluation order. The first matching transition wins.
    2. Verify the conditions in your ShouldTransition method match the actual runtime state. Use the Entity Inspector to check LocomotionContact, LocomotionVelocity, and your input component values.
    3. If using the inline EvaluateTransitions pattern, check the order of if checks - earlier checks have higher priority.

    Verify: Add a temporary log inside the transition's ShouldTransition to see which condition fails.

    Character launched unexpectedly

    Cause: An external impulse is applied from a dynamic body collision, or platform velocity is being applied twice.

    Fix:

    1. Enable impulse debug visualization to see where forces are coming from
    2. Check that you are not manually adding PlatformHorizontalVelocity in OnIntegrateState - the system handles platform inheritance automatically in the ComputeInheritedDisplacement step
    3. If dynamic body collisions are injecting too much velocity, check the Mass and MassToDynamic values on the Character Physics Profile

    Verify: Disable impulse handling temporarily by overriding HandleExternalImpulses to a no-op on your state machine struct.

    Character clips through or gets stuck on moving platforms

    Cause: The ground snapping distance is too small for the platform's speed.

    Fix:

    1. Override ComputeGroundSnappingDistance (or OverrideGroundSnappingDistance partial method) on your context to account for platform vertical velocity
    2. See the sample's LocomotionContextExtensions.cs for a reference implementation that adds platformDistance

    Verify: Enable path point debug visualization to see if the character loses ground contact on fast platforms.

    Source generator not producing code

    Cause: The source generator cannot find or process your state machine attributes.

    Fix:

    1. Ensure your state enum has the [LocomotionStateMachine] attribute
    2. Ensure each enum value has a [LocomotionState(typeof(YourState))] attribute
    3. Check that your project references the Wayn.Games.Locomotion.Runtime assembly
    4. Restart Unity or reimport the package - the source generator runs at compile time

    Verify: Check the Roslyn Analyzers section in your assembly definition to confirm the source generator DLL is referenced.

    Build compiles but character doesn't work in player

    Cause: Generic types used by the source generator may not be registered for AOT compilation.

    Fix:

    1. Ensure you have [assembly: RegisterGenericComponentType(typeof(LocomotionInputAsset<YourInput>))] in your assembly
    2. Check for Burst compilation errors in the build log - state methods must be Burst-compatible (no managed types)
    3. Verify there are no managed types (string, class, arrays) in state structs or context

    Verify: Build with Development Build enabled and check the Player log for Burst or ECS errors.

    Debug visualization not showing

    Cause: Debug components are not added or the Scene view is not visible.

    Fix:

    1. Select the character entity's authoring GameObject in the sub-scene
    2. Add a LocomotionDebugConfigAuthoring component
    3. Configure which visualizations to enable in the Inspector
    4. Enter Play mode and switch to the Scene view (debug gizmos do not render in Game view)

    Verify: Debug data is baked into a blob asset, so changes require re-entering Play mode.
