    La boucle d'exécution

    À chaque mise à jour fixe, le job de locomotion traite chaque entité personnage à travers une séquence précise d'étapes. Comprendre cette séquence est essentiel pour écrire une logique d'état correcte.

    Vue d'ensemble

    Le job de locomotion s'exécute comme un IJobChunk compilé par Burst planifié dans le FixedStepSimulationSystemGroup, après l'étape de physique. Chaque personnage est traité indépendamment et en parallèle.

    Les étapes s'exécutent dans cet ordre exact :

    flowchart TD
    START(["Début de frame"])

    subgraph S1["1. HandleExternalImpulses"]
        HEI["Accumuler les impulsions de vélocité<br/>depuis CharacterContactSystem"]
    end

    subgraph S2["2. Integrate - <code>OnIntegrateState</code>"]
        INT["Calculer les vélocités + déplacement"]
    end

    subgraph S3["3. SolveOverlaps"]
        SO["Dé-collision multi-plan<br/>Définit OverlapState: None | Push | Pinch | Crush"]
    end

    subgraph S4["4. CollideAndSlide - <code>OnResolveStateCollisions</code>"]
        CAS_CRUSH{"OverlapState<br/>== Crush?"}
        CAS_LOOP{"HasRemaining-<br/>Displacement &&<br/>bounce < MaxBounce?"}
        CAS["Projeter, caster, glisser,<br/>mettre à jour les contacts"]
        CAS_CRUSH -- Oui --> CAS_END
        CAS_CRUSH -- Non --> CAS_LOOP
        CAS_LOOP -- Oui --> CAS --> CAS_LOOP
        CAS_LOOP -- Non --> CAS_END
        CAS_END([terminé])
    end

    subgraph S5["5. SnapToGround"]
        SNAP_NONE{"ContactType<br/>== None?"}
        SNAP_UP{"En mouvement vers le haut?<br/><i>dot(vVel, gravity) < −ε</i>"}
        SNAP_VVEL{"VerticalVelocity<br/>== 0?"}
        SNAP["Capsule cast vers le bas<br/>Ajuster la position + mettre à jour le contact"]
        SNAP_NONE -- "Oui (passer)" --> SNAP_END
        SNAP_NONE -- Non --> SNAP_UP
        SNAP_UP -- "Oui (effacer le contact)" --> SNAP_END
        SNAP_UP -- Non --> SNAP_VVEL
        SNAP_VVEL -- "Non (passer)" --> SNAP_END
        SNAP_VVEL -- "Oui (au repos)" --> SNAP --> SNAP_END
        SNAP_END([terminé])
    end

    subgraph S6["6. InheritDisplacementFromMotionVelocity"]
        IH_GROUND{"Contact<br/>au sol?"}
        IH_CRUSH{"Écrasement?"}
        IH_DYN{"Debout sur un<br/>corps dynamique?"}
        IH["Calculer + appliquer le déplacement de plateforme<br/>Appliquer la rotation de torsion"]
        IH_GROUND -- Non --> IH_END
        IH_GROUND -- Oui --> IH_CRUSH
        IH_CRUSH -- Oui --> IH_END
        IH_CRUSH -- Non --> IH_DYN
        IH_DYN -- Non --> IH_END
        IH_DYN -- Oui --> IH --> IH_END
        IH_END([terminé])
    end

    subgraph S7["7. EvaluateTransition"]
        ET["Dispatch généré par le source<br/>vers les structs LocomotionTransition"]
    end

    DONE(["Fin de frame"])

    START --> HEI
    HEI --> INT
    INT --> SO
    SO --> CAS_CRUSH
    CAS_END --> SNAP_NONE
    SNAP_END --> IH_GROUND
    IH_END --> ET
    ET --> DONE

    Chaque étape du pipeline est une méthode d'interface par défaut (DIM) sur ILocomotionStateMachine qui peut être surchargée sur la struct partielle de votre machine à états. Voir Personnaliser le pipeline d'exécution pour les détails.

    Il n'y a pas de méthode OnUpdateState. Les phases d'intégration et de résolution de collision forment ensemble la "mise à jour" de votre état. Les transitions sont évaluées une seule fois, à la toute fin, après que toute la physique est résolue.

    Description étape par étape

    1. HandleExternalImpulses

    Le CharacterContactSystem s'exécute pendant le PhysicsSimulationGroup (entre la création des contacts et la création des jacobiens) et collecte les impulsions de vélocité provenant de sources externes, comme :

    • Les corps rigides dynamiques qui ont collisionné avec le personnage
    • D'autres personnages poussant contre celui-ci
    • Des explosions ou champs de force

    Ces impulsions sont appliquées à la vélocité du personnage avant que votre logique d'état ne s'exécute, donc votre OnIntegrateState voit la vélocité mise à jour. C'est pourquoi vous ne devriez pas effacer la vélocité inconditionnellement au début de l'intégration, sauf si vous avez l'intention d'ignorer les forces externes.

    2. Integrate (OnIntegrateState)

    C'est là que votre logique d'état calcule les vélocités et déplacements pour la frame actuelle. Le système appelle la méthode OnIntegrateState de votre état.

    Dans cette étape, vous devriez :

    • Calculer l'accélération à partir de l'entrée (direction de mouvement, force de saut)
    • Appliquer l'accélération de la gravité
    • Appeler baseContext.UpdateVelocitiesAndComputeDisplacements() pour convertir les vélocités en déplacements

    Ce que vous ne devriez PAS faire ici :

    • Déplacer directement la position du personnage (cela se produit à l'étape de collision)
    • Ajouter la vélocité de la plateforme (l'étape ComputeInheritedDisplacement gère cela automatiquement)
    • Lire les informations de contact de cette frame (les contacts n'ont pas encore été résolus ; vous voyez les contacts de la frame précédente)

    L'équation cinématique

    UpdateVelocitiesAndComputeDisplacements applique les équations cinématiques standard :

    displacement += velocity * dt + 0.5f * acceleration * dt * dt;
velocity += acceleration * dt;

    Ceci est fait séparément pour les composantes horizontale et verticale, permettant un comportement différent pour chacune (ex. la gravité n'affecte que le vertical, l'entrée de mouvement n'affecte que l'horizontal).

    3. SolveOverlaps

    Après l'intégration, le système vérifie si le collider du personnage chevauche actuellement un autre collider. Cela peut arriver quand :

    • Une plateforme mobile pousse le personnage dans un mur
    • Un autre personnage occupe le même espace
    • Le personnage a été téléporté dans la géométrie

    Le solveur de chevauchement tente de pousser le personnage hors du chevauchement. S'il ne peut pas (ex. le personnage est coincé entre deux surfaces mobiles), il suit l'état de chevauchement :

    OverlapState Signification
    None Aucun chevauchement détecté
    Push Le personnage a été poussé avec succès
    Pinch Le personnage est comprimé mais pas complètement piégé
    Crush Le personnage est complètement encastré sans échappatoire

    Vous pouvez lire ctx.BaseContext.OverlapState dans votre logique d'état pour réagir aux situations d'écrasement (ex. tuer le personnage).

    4. CollideAndSlide (OnResolveStateCollisions)

    Après que l'intégration a calculé le déplacement souhaité, le personnage doit effectivement se déplacer à travers le monde en respectant les colliders. C'est la boucle collide-and-slide.

    Le système invoque le OnResolveStateCollisions de votre état tant qu'il reste du déplacement, jusqu'à CharacterPhysicsProfile.MaxBounce itérations. Cette approche multi-rebond permet au personnage de glisser le long des murs, d'enjamber les obstacles et de naviguer dans une géométrie complexe.

    Comportement par défaut

    Si vous ne surchargez pas OnResolveStateCollisions, le DIM par défaut appelle baseContext.Move(), qui effectue deux opérations :

    1. Correction de déplacement - Ajuste HorizontalDisplacement et VerticalDisplacement en fonction du type de contact actuel :

      Type de contact Comportement
      None Le déplacement est appliqué tel quel (mouvement libre dans l'air)
      Ground Le déplacement vertical est annulé ; le déplacement horizontal suit la pente du sol
      Wall La détection de marche est tentée ; si ce n'est pas une marche, le déplacement est projeté le long de la surface du mur
      Ceiling Le déplacement vertical est projeté le long du plafond ; le déplacement horizontal est contraint

    2. Cast et déplacement - Appelle PhysicsOperations.MoveUpToNextObstacle pour caster la capsule vers l'avant du déplacement corrigé, déplacer le personnage au point d'impact (ou la distance complète si rien n'a été touché), et mettre à jour les informations de contact.

    Pour personnaliser la résolution des collisions, voir Réponse de collision personnalisée.

    Détection de marches

    Lorsqu'un contact avec un mur est détecté, le Move() par défaut vérifie s'il pourrait s'agir d'une marche :

    1. Le personnage doit être en contact avec une surface
    2. La pente doit dépasser MaxSlope
    3. Le personnage doit se déplacer vers la pente
    4. La hauteur du contact doit être inférieure à MaxStepHeight

    Si toutes les conditions sont remplies, PhysicsOperations.TryStep tente de déplacer le personnage par-dessus la marche.

    5. SnapToGround

    Après que toute la résolution de collision est terminée, le système effectue un capsule cast vers le bas pour détecter la surface du sol sous le personnage. Cela garantit que le personnage reste au sol lorsqu'il descend des pentes ou des escaliers, plutôt que de devenir aérien pendant une frame.

    La distance d'accrochage est calculée par la méthode ComputeGroundSnappingDistance de votre contexte. L'implémentation par défaut retourne une distance codée en dur de 1f. Surchargez cette méthode sur la struct partielle de votre contexte pour prendre en compte la hauteur des marches, l'angle de la pente, la vitesse ou la vélocité de la plateforme selon les besoins.

    Si une surface de sol est trouvée dans la distance d'accrochage, la position du personnage est ajustée vers le bas et les informations de contact sont mises à jour.

    6. InheritDisplacementFromMotionVelocity

    Si le personnage se tient sur un corps dynamique (cinématique ou entièrement simulé), cette étape calcule le déplacement et la rotation que le personnage hérite de la vélocité de mouvement de ce corps.

    Le déplacement est appliqué à la position du personnage dans la même frame. Les vélocités héritées sont stockées dans :

    • baseContext.PlatformHorizontalVelocity
    • baseContext.PlatformVerticalVelocity

    Celles-ci sont disponibles immédiatement - par exemple, le OnEnterState d'un état de saut (appelé pendant EvaluateTransition dans la même frame) peut lire PlatformHorizontalVelocity pour préserver l'élan en sautant d'une plateforme. L'ajustement de position dû au mouvement de la plateforme est appliqué automatiquement, c'est pourquoi vous ne devriez jamais ajouter la vélocité de la plateforme manuellement dans OnIntegrateState.

    7. EvaluateTransition

    La dernière étape évalue toutes les structs [LocomotionTransition] enregistrées pour l'état actuel. À ce stade :

    • Le personnage s'est déplacé à sa position finale
    • Tous les contacts sont résolus et à jour
    • L'héritage de plateforme est calculé

    Le générateur de source dispatche vers la méthode ShouldTransition de chaque transition dans l'ordre déterminé par les attributs [EvaluateBefore]/[EvaluateAfter]. La première transition correspondante l'emporte. Le générateur gère automatiquement l'appel de OnExitState sur l'état actuel et OnEnterState sur le nouvel état.
