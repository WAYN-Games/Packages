    Extending the Package

    The MGM Character Controller is designed to be extended without modifying the package source. This page covers common extension points and patterns.

    Extension Points

    1. Custom States

    The most common extension. Add new movement behaviors (wall running, swimming, climbing, dashing) by creating new state structs and adding them to your state enum.

    See Writing Custom States for the full guide.

    2. Custom Profiles

    Your ILocomotionProfile struct can contain any unmanaged data. Add fields for new gameplay parameters:

    [Serializable]
public partial struct MyProfile : ILocomotionProfile
{
    public MovingProfile Moving;
    public FallingProfile Falling;
    public JumpingProfile Jumping;

    // Custom extension
    public SwimmingProfile Swimming;
    public DashProfile Dash;
    public WallRunProfile WallRun;
}

    Each sub-profile is a plain struct with configuration fields. The data is baked into a blob asset, so add corresponding fields to your LocomotionProfileAsset and BuildBlobAsset method.

    3. Custom Context

    Your context struct can include additional computed properties and helper methods:

    public partial struct MyContext : ILocomotionContext<MyProfile, MyInput, MyStateData>
{
    public BaseLocomotionContext BaseContext;
    public MyStateData StateData;

    // Standard interface properties...

    // Custom helpers
    public bool IsUnderwater => BaseContext.CharacterPosition.Position.y < WaterLevel;
    public bool WantsToMove => Input.MovementMagnitude > math.EPSILON;
    public bool CanDash => StateData.DashCooldown <= 0;

    // Custom ground snap for swimming (no ground snap when submerged)
    public void ComputeGroundSnappingDistance(in MyProfile profile, out float distance)
    {
        if (IsUnderwater)
        {
            distance = 0; // No ground snapping while swimming
            return;
        }
        // Normal ground snap calculation...
    }
}

    4. Custom StateData

    Add per-state persistent data for any state-specific tracking:

    public partial struct MyStateData : IComponentData
{
    public JumpData JumpData;
    public float DashCooldown;
    public float WallRunTimer;
    public float3 WallNormal;  // Remembered from wall run entry
}

    State data persists across frames and state transitions. Clean it up in OnEnterState or OnExitState as appropriate.

    5. Custom Input

    Add input channels for new actions:

    public partial struct MyInput : ICharacterLocomotionInput<MyInput>
{
    public float3 Movement;
    public float JumpTrigger;
    public float DashTrigger;      // New action
    public float InteractTrigger;  // New action

    public void ApplyInputsToCharacter(in MyInput playerInputs)
    {
        Movement = playerInputs.Movement;
        JumpTrigger += playerInputs.JumpTrigger;       // Accumulate triggers
        DashTrigger += playerInputs.DashTrigger;
        InteractTrigger += playerInputs.InteractTrigger;
    }
}

    Remember to read the new inputs in your PlayerInputAsset subclass and consume triggers in your state logic.

    6. Custom Ground Snapping

    Override ComputeGroundSnappingDistance on your context to control when and how far the character snaps to the ground:

    public void ComputeGroundSnappingDistance(in MyProfile profile, out float distance)
{
    // Disable ground snapping while jumping upward
    if (StateData.CurrentState == MyStates.Jumping &&
        math.dot(BaseContext.VerticalVelocity, -BaseContext.GravityDirection) > 0)
    {
        distance = 0;
        return;
    }

    // Default calculation
    var slopeDistance = math.length(BaseContext.HorizontalVelocity * BaseContext.TimeStep) *
                       math.tan(math.radians(CharacterPhysicsProfile.MaxSlope));
    distance = math.max(CharacterPhysicsProfile.MaxStepHeight, slopeDistance);
}

    7. Custom Collision Response

    Override OnResolveStateCollisions to add custom behavior when the character hits something:

    public void OnResolveStateCollisions(ref MyContext ctx)
{
    BaseLocomotionContext baseContext = ctx.BaseContext;
    baseContext.Move();

    // Custom: bounce off trampolines
    if (baseContext.IsGroundingContact &&
        HasTrampolineTag(baseContext.LocomotionContact.ValueRO.Entity))
    {
        baseContext.VerticalVelocity = -baseContext.VerticalVelocity * 1.5f;
    }

    ctx.BaseContext = baseContext;
}

    8. Custom Systems

    You can add systems that modify locomotion components at any point in the pipeline:

    [UpdateInGroup(typeof(FixedStepSimulationSystemGroup))]
[UpdateBefore(typeof(MyLocomotionSystem))]
partial struct GravityZoneSystem : ISystem
{
    public void OnUpdate(ref SystemState state)
    {
        foreach (var gravity in SystemAPI.Query<RefRW<LocomotionGravity>>()
            .WithAll<InGravityZone>())
        {
            gravity.ValueRW.Gravity = new float3(0, -4.9f, 0); // Half gravity
        }
    }
}

    Customizing the Execution Pipeline

    Each step in the execution loop is a default interface method (DIM) on ILocomotionStateMachine. You can override any of them on your state machine's partial struct to replace or wrap the default behavior.

    The source generator produces a partial struct for your state machine (e.g., MyStateMachine). Because it is partial, you can add a second partial declaration that overrides any DIM:

    public partial struct MyStateMachine
{
    public void Integrate(ref MyStateMachine stateMachine, ref MyContext context)
    {
        // Skip movement entirely when crushed
        if (context.BaseContext.OverlapState == OverlapState.Crush)
            return;

        // Otherwise run the default per-state integration
        stateMachine.IntegratePerState(ref context);
    }
}

    Overridable Pipeline Methods

    Method Default Behavior Override to...
    Execute Orchestrates the full pipeline (all steps below in order) Replace the entire execution loop
    HandleExternalImpulses Accumulates velocity impulses from body contacts Custom impulse filtering or scaling
    Integrate Calls IntegratePerState (skips if crushed) Add pre/post integration logic
    SolveOverlaps Multi-plane de-collision algorithm Custom overlap resolution
    CollideAndSlide Loops ResolveCollisionPerState up to MaxBounce Custom collision iteration
    SnapToGround Downward capsule cast to maintain ground contact Custom ground adhesion
    InheritDisplacementFromMotionVelocity Computes platform velocity inheritance from physics bodies Custom platform riding behavior
    EvaluateTransition Calls EvaluateTransitionPerState Add pre/post transition logic
    Note

    IntegratePerState, ResolveCollisionPerState, and EvaluateTransitionPerState are generated by the source generator (switch dispatches to per-state methods). They are not intended to be overridden directly.

    See The Execution Loop for the ordering and data flow between these steps.

    Customizing the Locomotion System

    The source generator creates a partial interface I{Prefix}LocomotionSystem that inherits from ILocomotionSystem<...>. The generated system struct implements this interface with DIM lifecycle methods that you can override.

    To customize, add your own partial interface declaration:

    public partial interface IMyLocomotionSystem
{
    void ILocomotionSystem<MyStateMachine, MyStates, MyContext, MyProfile, MyInput, MyStateData>
        .OnLocomotionSystemUpdate(
            ref LocomotionJobData jobData, ref EntityQuery query, ref SystemState state)
    {
        if (query.IsEmptyIgnoreFilter) return;
        jobData = jobData.Update(ref state);

        // Schedule single-threaded instead of parallel (useful for debugging)
        state.Dependency = ScheduleLocomotionJob(
            state.Dependency,
            state.WorldUnmanaged.Time.DeltaTime,
            jobData, query,
            jobData.GetPhysicsWorldSingleton());
    }
}

    Overridable System Lifecycle Methods

    Method Default Behavior Override to...
    OnLocomotionSystemCreate Builds entity query and creates LocomotionJobData Add RequireForUpdate, custom query filters, rate manager setup
    OnLocomotionSystemUpdate Updates job data and calls ScheduleParallelLocomotionJob Use single-threaded scheduling, add conditional skip, integrate rate manager
    OnLocomotionSystemDestroy No-op Custom resource cleanup
    Note

    The default DIM implementations use ref parameters instead of SystemAPI because Unity's source generator cannot patch call sites in code emitted by the package's source generator (both run in parallel). Your own partial interface overrides can use SystemAPI since they are visible to Unity's generator.

    DIM Customization Reference

    Complete reference of all default interface methods across the package:

    Interface Method Default Override to...
    ILocomotionState OnEnterState no-op Initialize state-specific data on entry
    ILocomotionState OnIntegrateState Apply gravity + compute displacements Custom movement logic per state
    ILocomotionState OnExitState no-op Clean up state-specific data on exit
    ILocomotionState OnResolveStateCollisions baseContext.Move() Custom collision response per state
    ILocomotionContext ComputeGroundSnappingDistance Returns 1f Speed-dependent or state-dependent snap distance
    ILocomotionStateMachine Execute Full pipeline Replace the entire execution loop
    ILocomotionStateMachine HandleExternalImpulses Accumulate contact impulses Custom impulse handling
    ILocomotionStateMachine Integrate Call IntegratePerState Pre/post integration logic
    ILocomotionStateMachine SolveOverlaps Multi-plane de-collision Custom overlap resolution
    ILocomotionStateMachine CollideAndSlide Bounce loop up to MaxBounce Custom collision loop
    ILocomotionStateMachine SnapToGround Downward capsule cast Custom ground adhesion
    ILocomotionStateMachine InheritDisplacementFromMotionVelocity Platform velocity inheritance Custom platform riding
    ILocomotionStateMachine EvaluateTransition Call EvaluateTransitionPerState Pre/post transition logic
    ILocomotionSystem OnLocomotionSystemCreate Build query + job data Custom system initialization
    ILocomotionSystem OnLocomotionSystemUpdate Schedule parallel Custom scheduling strategy
    ILocomotionSystem OnLocomotionSystemDestroy no-op Custom resource cleanup
    ICharacterLocomotionInput ApplyInputsToCharacter no-op Accumulate/merge inputs between fixed updates

    Multiple State Machines

    You can have multiple independent state machine types in the same project. For example:

    • GroundLocomotionStates - Walking, running, jumping, falling
    • VehicleStates - Driving, braking, boosting, airborne
    • SwimmingStates - Surface, submerged, diving

    Each has its own enum, context, profile, input, state data, and generated systems. They are completely independent.

    To switch a character between state machines, you would need to add/remove the relevant components (the state machine component, input, profile, state data). This is an advanced pattern that involves structural entity changes.

    Working with the Source Generator

    The Roslyn source generator reads attributes on your anchor struct and state structs and generates:

    1. State machine component with dispatch logic
    2. Input reader system
    3. ISystem lifecycle methods (OnCreate, OnUpdate, OnDestroy) on your locomotion system

    The generator source is at SourceGenerator/SrcGenProj~/. To modify the generated code patterns:

    1. Edit StateMachineSourceGenerator.cs
    2. Build with dotnet build from the SrcGenProj~ directory
    3. The DLL is copied to SourceGenerator/ and picked up by Unity
    Warning

    Modifying the source generator affects all state machines in the project. Test thoroughly.

    Creating Composite Characters

    For characters that need both ECS locomotion and complex visual hierarchies, use the CompositeEntityAuthoring pattern:

    1. The Locomotor entity carries all physics and state machine components
    2. A Model entity follows the Locomotor via CharacterFollower
    3. The Model's GameObject can have child objects with Animators, particle systems, audio sources

    The CharacterFollowerAuthoring links the model to the locomotor during baking. At runtime, the CharacterFollowerSystem syncs the transform.

    Performance Considerations

    Scaling to Many Characters

    The locomotion job runs in parallel across chunks. To maximize throughput:

    • Keep all characters with the same archetype (same component set)
    • Use shared colliders (avoid ForceUniqueColliderAuthoring unless needed)
    • Profile shared blob assets (all characters referencing the same profile share memory)
    • Only enable debug visualization on characters you are actively investigating

    Burst Compilation

    State methods run inside a Burst-compiled job, so they must be Burst-compatible. Common mistakes that prevent Burst compilation:

    • Using string or List<T> types
    • Calling managed virtual methods (DIM lifecycle methods are dispatched by the source generator, not via virtual dispatch)
    • Accessing GameObject or MonoBehaviour APIs
    • Using Debug.Log (use [Conditional("ENABLE_UNITY_COLLECTIONS_CHECKS")] guards)

    Profiling

    Use Unity's Profiler to identify bottlenecks:

    • The locomotion job appears as LocomotionJob in the Jobs timeline
    • Per-stage profiler markers are static fields on ILocomotionStateMachine, gated by [Conditional("ENABLE_LOCOMOTION_PROFILER")] via ProfilerMarkerExtensions
    • The LocomotionProfilerHUD shows frame-level timing in-game
