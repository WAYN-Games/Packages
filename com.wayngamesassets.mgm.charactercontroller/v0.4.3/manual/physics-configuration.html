    Physics Configuration

    The CharacterPhysicsProfile defines the physical properties of the character's collision capsule and how it interacts with the world. This page explains each parameter, its effect, and how to tune it.

    CharacterPhysicsProfile Fields

    Capsule Geometry

    Field Type Description
    CapsuleHeight float Total height of the collision capsule from bottom to top, in world units. The capsule's bottom sphere sits at y = CapsuleRadius and the top sphere at y = CapsuleHeight - CapsuleRadius.
    CapsuleRadius float Radius of both hemisphere caps. Also determines the minimum width of the character. Default: 0.5.

    The capsule geometry directly affects:

    • Which gaps the character can fit through
    • How close the character can get to walls
    • Step detection (contact height relative to capsule bottom)

    Mass and Forces

    Field Type Description
    Mass float Weight of the character. Used for collision response calculations when the character collides with dynamic rigid bodies. Higher mass means the character pushes objects more.
    MassToDynamic float Multiplier applied to all impulses the character exerts on dynamic bodies during collision response.

    Movement Limits

    Field Type Description
    MaxSlope float Maximum slope angle (in degrees) the character can walk up. Surfaces steeper than this are treated as walls. Typical values: 45-60 degrees.
    MaxStepHeight float Maximum height (in world units) of obstacles the character can automatically step over. Typical values: 0.3-0.5.
    MaxBounce int Maximum number of collide-and-slide iterations per frame. Each bounce allows the character to redirect along a surface it hits. Higher values handle more complex geometry (corners, narrow corridors) but cost more. Default: 3.

    Collision Layers

    Field Type Description
    BelongsTo LayerMask The physics layer(s) this character belongs to. Other colliders check this to decide if they can collide with the character.
    CollidesWith LayerMask The physics layer(s) this character tests against. The character only detects collisions with colliders on these layers.

    To make characters ignore each other, put them on a layer that does not appear in other characters' CollidesWith mask. To make characters collide with each other, include the character layer in both masks.

    Collision Tolerance

    Field Type Description
    CollisionTolerance float Skin thickness for all collision queries. This is the distance at which the system considers two surfaces "in contact." Decrease for tighter collision response; increase if you observe jitter or tunneling. Defaults to CollisionWorld.DefaultCollisionTolerance.

    Interpolation

    Field Type Description
    InterpolationMode enum Interpolated enables Unity.Physics graphical smoothing between fixed updates, producing smoother visual movement. NotInterpolated updates the transform directly each fixed step, which can appear jittery at low fixed rates but is simpler to debug.

    Default Values

    The static CharacterPhysicsProfile.Default property provides sensible starting values:

    CharacterPhysicsProfile.Default = new CharacterPhysicsProfile
{
    MaxBounce = 3,
    CapsuleRadius = 0.5f,
    CapsuleHeight = 2f,
    BelongsTo = 1,                                    // Layer 0 (Default)
    CollidesWith = ~0,                                 // All layers
    CollisionTolerance = CollisionWorld.DefaultCollisionTolerance
};

    Collision Response

    The character's capsule collider is created with CollisionResponsePolicy.CollideRaiseCollisionEvents. This means:

    1. The collider participates in collision detection
    2. Collision events are raised, allowing CharacterContactSystem to intercept contacts

    However, CollideRaiseCollisionEvents alone would still let Unity.Physics solve the contact normally. The package prevents this by setting JacobianFlags.Disabled on every contact that involves a character inside CharacterContactSystem. This disables the physics solver's constraint for that contact, giving the locomotion job full control over the character's movement. The system then records the contact data (penetration depth, surface normal, velocity) for use by the locomotion pipeline's SolveOverlaps and HandleExternalImpulses steps.

    Shared vs Unique Colliders

    By default, capsule colliders are shared across entities with identical geometry and filter settings. This saves memory when you have many characters with the same shape.

    If you need a unique collider per entity (e.g., for runtime shape modification), add a ForceUniqueColliderAuthoring component to the character's authoring GameObject.

    Tuning Guide

    Character walks through walls

    • Increase MaxBounce (try 8-10)
    • Check CollidesWith includes the wall's layer
    • Increase CollisionTolerance slightly

    Character jitters on the ground

    • Enable InterpolationMode.Interpolated
    • Check that CollisionTolerance is not too small
    • Verify the ground collider is static (not rebuilding each frame)

    Character cannot climb steps

    • Increase MaxStepHeight
    • Ensure the step geometry is clean (not sloped mesh artifacts)
    • Check that MaxSlope is high enough that the step face is classified as a wall

    Character slides off slopes it should walk on

    • Increase MaxSlope (the threshold between "ground" and "wall")

    Character clips through fast descending platforms

    • The ground snapping distance may be too small for platforms moving downward faster than the snap can track. Override ComputeGroundSnappingDistance in your context to account for higher vertical platform speeds

    How the Collider Is Built

    During baking, the BaseCharacterLocomotorBaker constructs the capsule collider from your profile:

    1. Creates CapsuleGeometry from CapsuleHeight and CapsuleRadius
    2. Creates CollisionFilter from BelongsTo and CollidesWith
    3. Sets CollisionResponsePolicy.CollideRaiseCollisionEvents
    4. Hashes the geometry + filter to check if a matching blob already exists
    5. If not, creates a new CapsuleCollider blob asset
    6. Attaches it as a PhysicsCollider component

    Gravity

    The LocomotionGravity Component

    Gravity is stored per-entity in the LocomotionGravity component, initialized to PhysicsStep.Default.Gravity (typically (0, -9.81, 0)) during baking. The component provides two computed properties:

    • GravityDirection - Normalized direction vector (usually (0, -1, 0))
    • GravityStrength - Magnitude (usually 9.81)

    You can modify the LocomotionGravity component at runtime to change gravity per-character. See Recipe: Per-Character Gravity.

    How Gravity Affects the Pipeline

    Gravity is not just a force applied during integration - it defines the character's entire reference frame. The GravityDirection determines what "up", "horizontal", and "vertical" mean for every step of the locomotion pipeline:

    Character orientation - At the start of each frame, the locomotion job reorients the character's rotation so that "up" is opposite the gravity direction. This happens before any state logic runs.

    Velocity decomposition - The character's velocity is split into HorizontalVelocity (perpendicular to gravity) and VerticalVelocity (along gravity). All state logic works with these decomposed values, not raw world-space velocity.

    Contact classification - Surfaces are classified as Ground, Wall, or Ceiling based on the angle between their normal and the gravity direction:

    • Ground: normal-to-gravity angle below MaxSlope
    • Wall: angle above MaxSlope but not facing away from gravity
    • Ceiling: normal points in the same general direction as gravity

    Displacement correction - The Move() method projects displacement along surfaces relative to gravity. Ground contact cancels vertical displacement; wall contact projects horizontal displacement along the wall plane defined relative to gravity.

    Capsule orientation - All physics casts (collide-and-slide, snap-to-ground, step detection) position the capsule along the gravity axis, not world up.

    Snap-to-ground - The downward capsule cast is performed along GravityDirection, not (0, -1, 0).

    Platform inheritance - Platform velocity is split into horizontal and vertical components relative to the contact surface normal.

    Input and Gravity

    Movement input should be projected onto the plane perpendicular to gravity. The sample's MovingState demonstrates this:

    float3 forwardOnPlane = movementDirection - math.projectsafe(movementDirection, -baseContext.GravityDirection);
forwardOnPlane = math.normalizesafe(forwardOnPlane);

    This ensures "forward" always means "forward along the ground" regardless of gravity direction.

    Using Gravity in State Logic

    Your state's OnIntegrateState reads gravity from the context to compute vertical acceleration:

    float3 gravityAcceleration = baseContext.GravityDirection * baseContext.GravityStrength;
baseContext.UpdateVelocitiesAndComputeDisplacements(moveAcceleration, gravityAcceleration);

    Jump force is applied opposite to gravity:

    baseContext.VerticalVelocity = jumpForce * -baseContext.GravityDirection;

    Because the entire pipeline is gravity-relative, changing LocomotionGravity at runtime automatically adapts contact classification, displacement projection, capsule orientation, and ground snapping - no state logic changes needed.
