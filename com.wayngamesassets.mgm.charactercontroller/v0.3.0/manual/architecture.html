    Architecture Overview

    This page explains the high-level design of the MGM Character Controller: the entity types, the role of each layer, and how data flows through the system.

    Design Philosophy

    The package is built around three principles:

    1. Performance first - All locomotion logic runs in Burst-compiled parallel jobs. No managed allocations on the hot path.
    2. Separation of concerns - Physics simulation (ECS) is decoupled from presentation (GameObjects). You can run thousands of characters without any GameObjects at all.
    3. Extensibility through composition - Instead of a monolithic character controller, you compose behavior from a state machine, a physics profile, an input component, and a context, all of which are user-defined types.

    Entity Types

    The package defines three distinct entity roles. Understanding them is essential.

    Locomotor (Pure ECS)

    The Locomotor is the primary character entity. It lives entirely in ECS and carries all the components needed for physics-based movement:

    Component Purpose
    LocalTransform World position and rotation
    PhysicsCollider Capsule collider for collision detection
    PhysicsVelocity Required by Unity.Physics (mostly zeroed; the package drives movement directly)
    LocomotionVelocity The character's actual velocity, maintained by the state machine
    LocomotionGravity Gravity vector applied to this character
    LocomotionContact Current surface contact information (normal, type, entity)
    CharacterPhysicsProfileBlob Baked physics configuration (capsule size, mass, slopes)
    LocomotionProfileBlob<TProfile> Baked gameplay profile (speeds, jump force, etc.)
    TStateMachine Source-generated state machine component (current state + dispatch)
    TInput User-defined input component
    TStateData User-defined per-state persistent data

    The Locomotor is a kinematic body. Unlike Unity.Physics dynamic bodies, it does not respond to forces automatically. Instead, the locomotion job computes displacement each frame and moves the collider through the world using a collide-and-slide algorithm.

    Model (Hybrid ECS + GameObject)

    The Model is an optional entity that provides visual representation. It carries a reference to a GameObject prefab that gets instantiated at runtime. The model entity follows the Locomotor entity via the CharacterFollower component.

    Why separate them? Because:

    • Locomotion logic runs in parallel Burst jobs that cannot touch GameObjects
    • Visual updates (animation, particles, audio) need MonoBehaviour access
    • You can run headless simulations (server, AI training) with zero GameObjects

    Player (Input Source)

    The Player entity represents an input source. It owns a ControlledCharacters buffer that lists which Locomotor entities it controls. A Player also has:

    • A PlayerGameObjectPrefab referencing a GameObject with camera, input system, and UI
    • A PlayerGameObjectInstance linking to the instantiated runtime copy
    • An LocomotionInputAsset<TInput> that reads hardware input and maps it to the input component

    Players are managed by the PlayerGOManagerSystem, which instantiates and destroys their GameObjects as players join and leave.

    Assembly Structure

    The package is split into three runtime assemblies:

    Wayn.Games.Locomotion.Runtime     Core ECS logic (state machine, physics, components, systems)
Wayn.Games.Locomotion.Hybrid      GameObject bridge (transform sync, entity-GO mapping)
Wayn.Games.Locomotion.Editor      Editor tools (scaffolding, inspectors)

    Your game code references Runtime and optionally Hybrid. The Editor assembly is editor-only and provides scaffolding menus.

    Source Generation

    A Roslyn source generator reads your state enum annotated with [LocomotionStateMachine] and [LocomotionState] attributes and produces:

    1. State machine component - An IComponentData struct that holds the current state and dispatches to the correct state implementation
    2. Locomotion system - An ISystem that schedules the LocomotionJob each fixed update
    3. Input reader system - An ISystem that reads input from the PlayerInputAsset and writes it to the TInput component
    4. Player input manager - Manages per-player input asset instances

    This means you never write scheduling code, component type handle boilerplate, or state dispatch switch statements. You define your states, annotate your enum, and the generator handles the rest.

    Blob Assets

    Configuration data (physics profiles, locomotion profiles, debug config) is stored as blob assets rather than regular components. Blob assets are:

    • Immutable at runtime (safe for parallel access)
    • Shared across entities with identical configuration (memory efficient)
    • Created during baking from ScriptableObject authoring assets

    This means changing a profile value requires re-baking (exiting and re-entering play mode, or modifying the asset in the editor).
