    Hybrid GameObject Layer

    The Hybrid layer bridges the gap between ECS entities and GameObjects. It enables characters driven by Burst-compiled ECS logic to have animated 3D models, particle effects, audio sources, and cameras, all of which require MonoBehaviour and GameObject APIs.

    Why a Hybrid Layer?

    Burst-compiled jobs cannot touch GameObjects, Cameras, Animators, or any managed Unity API. But games need these for rendering, animation, audio, and UI. The hybrid layer solves this by:

    1. Running all locomotion logic in pure ECS (fast, parallel, Burst-compatible)
    2. Syncing entity transforms to GameObjects each frame (main thread, after jobs complete)
    3. Letting MonoBehaviours on those GameObjects react to position/rotation changes

    This means you get the performance of ECS for simulation and the convenience of GameObjects for presentation.

    Assembly

    The hybrid layer lives in the Wayn.Games.Locomotion.Hybrid assembly, which references the Runtime assembly. Your game code can reference Hybrid if it needs to interact with the bridge systems.

    Core Types

    EntityGameObject (MonoBehaviour)

    Attached to a GameObject that is linked to an ECS entity. It stores the Entity reference and registers/unregisters the GameObject's Transform with the singleton transform sync system.

    When the GameObject is destroyed, it automatically unregisters from the sync system to prevent dangling references.

    EntityGameObjectInstance (MonoBehaviour)

    A per-instance marker that links a specific GameObject instance to its entity. Used when the same prefab is instantiated multiple times for different entities.

    GameObjectEntity (Component)

    An ECS component that stores the UnityObjectRef<GameObject> reference, allowing systems to look up the GameObject from the entity side.

    EntityGoCleanup (Component)

    A cleanup component added to entities whose associated GameObjects need to be destroyed. The EntityGoCleanupSystem processes these to ensure GameObjects are properly disposed when their entities are destroyed.

    Systems

    EntityGoInitialisationSystem

    Runs when new entities with GameObjectEntity components appear. It:

    1. Instantiates the referenced GameObject prefab
    2. Attaches an EntityGameObject MonoBehaviour to link it back to the entity
    3. Registers the transform for syncing

    EntityGoTransformSyncSystem

    A singleton system that maintains a registry of (Entity, Transform) pairs. Each frame, after all ECS jobs complete, it copies LocalTransform from each registered entity to the corresponding GameObject's Transform.

    This runs on the main thread because Transform access requires it, but the actual locomotion simulation that produces the positions ran in parallel Burst jobs.

    EntityGoCleanupSystem

    Processes entities with the EntityGoCleanup component. When an entity is destroyed, this system ensures the associated GameObject is also destroyed, preventing orphaned GameObjects in the scene.

    EntityGoPreviewSystem (Editor Only)

    Provides editor-time preview of entity-linked GameObjects. This allows you to see the visual representation in the Scene view without entering Play mode.

    Character Follower

    The CharacterFollower component is an ECS component added to "model" entities. It stores a reference to the Locomotor entity that the model should follow.

    The CharacterFollowerSystem runs each frame and copies the Locomotor's position and rotation to the model entity. The model entity then has its transform synced to the associated GameObject by EntityGoTransformSyncSystem.

    This two-hop chain (Locomotor -> Model Entity -> GameObject) ensures:

    • The Locomotor runs in parallel with no GameObject dependencies
    • The Model entity can apply visual offsets or smoothing
    • The GameObject receives the final visual transform

    Player GameObject Management

    PlayerGOManagerSystem

    This system manages the lifecycle of player GameObjects:

    1. Spawn: When a new Player entity appears with a PlayerGameObjectPrefab but no PlayerGameObjectInstance, the system instantiates the prefab and stores the instance reference
    2. Destroy: When a Player entity is destroyed, the system destroys the associated GameObject

    The player GameObject typically contains:

    • A Camera component
    • A CinemachineBrain for camera blending
    • A PlayerInput component for input handling
    • A PlayerGoAuthoring with references to the camera and camera target

    CameraRigFollowControlledCharacterSystem

    Ensures the player's camera rig follows the controlled character. It reads the ControlledCharacters buffer and updates the camera target position to match the active character's position.

    DestructionManager

    A MonoBehaviour that handles cleanup when a hybrid GameObject is destroyed. It ensures all entity references are properly cleaned up and prevents errors from accessing destroyed GameObjects.

    LocomotionProfilerHUD

    An optional MonoBehaviour that displays runtime profiling information as an in-game overlay. Useful during development to monitor the performance of the locomotion system.

    Setting Up a Hybrid Character

    To create a character with a visual model:

    1. Create the Locomotor entity (in sub-scene)

    Add your Character Locomotor Authoring component as described in Getting Started.

    2. Create the Model (in sub-scene)

    1. Add a child GameObject under (or near) the Locomotor with your visual mesh/model
    2. Add a CharacterFollowerAuthoring component
    3. Set the Locomotor field to reference the Locomotor's authoring component
    4. The model can have an Animator, AudioSource, particle systems, etc.

    3. Create the Player prefab (regular GameObject, NOT in sub-scene)

    1. Create a GameObject with Camera, CinemachineBrain, PlayerInput
    2. Add a PlayerGoAuthoring component with camera references
    3. Save it as a prefab

    4. Create the Player entity (in sub-scene)

    1. Create a GameObject in the sub-scene
    2. Add your Player Character Authoring component
    3. Assign the character entity, input asset, and player GO prefab

    At runtime:

    1. The Player entity is baked, triggering PlayerGOManagerSystem to instantiate the player prefab
    2. The input asset is initialized with the player's camera and input references
    3. The model entity follows the Locomotor via CharacterFollowerSystem
    4. The model's GameObject transform is synced via EntityGoTransformSyncSystem
