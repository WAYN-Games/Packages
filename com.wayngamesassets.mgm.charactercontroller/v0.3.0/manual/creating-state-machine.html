    Creating a State Machine

    This guide explains how to create a complete locomotion state machine from scratch, either using the editor scaffolding tool or by hand.

    Using the Editor Tool (Recommended)

    The fastest way to create a state machine is through the Unity editor menu:

    1. Right-click in the Project window where you want the state machine folder
    2. Select Assets > Create > WAYN Games > Locomotion > StateMachine
    3. Type a name for your state machine (e.g., Vehicle, Swimming, Flying)
    4. Press Enter

    The tool generates all required files with proper naming, namespaces, and wiring. You can immediately start customizing the generated code.

    Understanding What Gets Generated

    When you create a state machine named MyCharacter, the tool generates these types:

    The State Enum

    [LocomotionStateMachine("MyCharacterStateMachine",
    typeof(MyCharacterContext),
    typeof(MyCharacterProfile),
    typeof(MyCharacterInput),
    typeof(MyCharacterStateData))]
public enum MyCharacterStates : byte
{
    [LocomotionState(typeof(MyCharacterState))]
    MyCharacter
}

    This enum is the heart of the state machine. The [LocomotionStateMachine] attribute tells the source generator:

    • Name: What to call the generated state machine component and systems
    • Context: The struct that carries all data through the state lifecycle
    • Profile: The struct holding gameplay configuration (speeds, forces)
    • Input: The struct receiving player or AI commands
    • StateData: The struct holding per-state persistent data (jump counts, timers)

    Each enum value has a [LocomotionState] attribute linking it to a concrete state struct.

    The Context

    public struct MyCharacterContext : ILocomotionContext<MyCharacterProfile, MyCharacterInput, MyCharacterStateData>
{
    public BaseLocomotionContext BaseContext { get; set; }
    public CharacterPhysicsProfile CharacterPhysicsProfile { get; set; }
    public MyCharacterProfile Profile { get; set; }
    public MyCharacterInput Input { get; set; }
    public MyCharacterStateData StateData { get; set; }
}

    The context is a data bag that is passed by reference through every state lifecycle method. It wraps:

    • BaseLocomotionContext - The core physics state (position, velocity, displacement, contacts, gravity)
    • CharacterPhysicsProfile - The baked capsule/physics configuration
    • Your custom profile, input, and state data
    Tip

    If you frequently modify BaseContext fields, declare it as a public field rather than using the auto-property. This lets you use ref access: ref BaseLocomotionContext baseContext = ref ctx.BaseContext;

    The Profile

    [Serializable]
public struct MyCharacterProfile : ILocomotionProfile
{
    // Add your configuration fields here:
    // public float MaxSpeed;
    // public float JumpForce;
}

    Profile data is baked into a blob asset during entity conversion. It is immutable at runtime and shared across all entities that reference the same profile asset. Put all tunable gameplay parameters here.

    The Input

    public struct MyCharacterInput : ICharacterLocomotionInput<MyCharacterInput>
{
    // Add your input fields here:
    // public float3 Movement;
    // public float JumpTrigger;

    public void ApplyInputsToCharacter(in MyCharacterInput playerInputs)
    {
        // Copy or accumulate input values here
    }
}

    The input struct is an ECS component. It receives values from the PlayerInputAsset at the Update rate and accumulates them for the fixed update. The ApplyInputsToCharacter method is called every Update frame by the generated input reader system.

    Warning

    Trigger inputs (like jump) must be accumulated, not overwritten. If the Update rate is higher than the fixed update rate, a trigger press could be missed. Accumulate with JumpTrigger += playerInputs.JumpTrigger and consume it in your state logic by resetting it to zero.

    The StateData

    public struct MyCharacterStateData : IComponentData
{
    // Add per-state persistent data here:
    // public int JumpCount;
    // public float StateTimer;
}

    State data persists across frames and state transitions. Unlike the context (which is rebuilt each frame from components), state data is stored as an ECS component and survives between frames.

    The Profile Asset

    [CreateAssetMenu(...)]
public class MyCharacterProfileAsset : LocomotionProfileAsset<MyCharacterProfile>
{
    // Serialized fields matching your profile struct

    public override ref MyCharacterProfile BuildBlobAsset(ref BlobBuilder blobBuilder)
    {
        ref MyCharacterProfile profile = ref blobBuilder.ConstructRoot<MyCharacterProfile>();
        // Copy serialized fields to blob
        return ref profile;
    }
}

    This ScriptableObject is what you assign in the Inspector. It serializes your profile data and builds the blob asset during baking.

    The Input Asset

    [CreateAssetMenu(...)]
public class MyCharacterPlayerInputAsset : PlayerInputAsset<MyCharacterInput>
{
    public override void ReadInputsFromAsset(out MyCharacterInput inputs)
    {
        // Read from Input System and populate inputs
    }

    public override void InitInputAsset(PlayerGameObjectInstance instance)
    {
        // Cache camera, PlayerInput component, etc.
    }
}

    The input asset reads hardware input and converts it to your input struct. It is a ScriptableObject that gets instantiated per player at runtime, so each player can cache their own camera and input device references.

    The Authoring Components

    Two authoring MonoBehaviours are generated:

    1. Character Locomotor Authoring - Placed on the character entity in the sub-scene. References the profile asset and configures the physics profile.
    2. Player Character Authoring - Placed on the player entity. References the character to control and the input asset.

    The Locomotion System

    public partial struct MyCharacterLocomotionSystem : ILocomotionSystem<
    MyCharacterStates, MyCharacterContext, MyCharacterProfile,
    MyCharacterInput, MyCharacterStateData, MyCharacterStateMachine>, ISystem
{
    // Implement OnCreateLocomotionSystem, OnUpdateLocomotionSystem,
    // and OnDestroyLocomotionSystem to hook into system lifecycle
}

    Your system implements the ILocomotionSystem interface, which provides the bridge between your types and the generated LocomotionJob. The source generator supplies the scheduling and dispatch logic.

    What the Source Generator Produces

    In addition to the files you write, the Roslyn source generator produces (at compile time, invisible in the Project window):

    1. State machine component (MyCharacterStateMachine) — An IComponentData struct with a CurrentState field and dispatch methods that call the correct state implementation based on the enum value
    2. Input reader system — Reads from the PlayerInputAsset each Update frame and writes to the TInput component
    3. ISystem lifecycle methods — Generated OnCreate, OnUpdate, and OnDestroy on your locomotion system struct that schedule the LocomotionJob and manage type handles

    You never need to write or maintain this code. The generator reads your enum and produces it automatically.

    Adding a New State

    After the initial generation, use the editor menu to add states:

    1. Right-click in the Project window
    2. Select the generated menu item for your state machine (e.g., Assets > Create > WAYN Games > Locomotion > MyCharacter > State)
    3. Name the state (e.g., Swimming)

    This generates a new state struct file and automatically adds it to your state enum with the [LocomotionState] attribute.

    Alternatively, create the state struct manually and add the enum entry yourself. See Writing Custom States for the struct template.

    Manual Creation

    If you prefer to create everything by hand (or need to understand what the tools do), follow this checklist:

    1. Create a States enum with [LocomotionStateMachine] attribute
    2. Create a context struct implementing ILocomotionContext<TProfile, TInput, TStateData>
    3. Create a profile struct implementing ILocomotionProfile
    4. Create an input struct implementing ICharacterLocomotionInput<TInput>
    5. Create a state data struct implementing IComponentData
    6. Create at least one state struct implementing ILocomotionState<TStates, TContext, TProfile, TInput, TStateData>
    7. Add [LocomotionState(typeof(YourState))] to each enum value
    8. Create a LocomotionProfileAsset<TProfile> ScriptableObject
    9. Create a PlayerInputAsset<TInput> ScriptableObject
    10. Create a BaseCharacterLocomotorAuthoring<TProfile> subclass with its Baker
    11. Create a BasePlayerCharacterAuthoring subclass
    12. Create a system implementing ILocomotionSystem<...> and ISystem
    13. Register the generic input component: [assembly: RegisterGenericComponentType(typeof(LocomotionInputAsset<TInput>))]

    The editor tool does all of this in one click, which is why it is recommended.
