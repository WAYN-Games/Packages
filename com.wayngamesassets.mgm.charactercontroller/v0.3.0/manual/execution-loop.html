    The Execution Loop

    Every fixed update, the locomotion job processes each character entity through a precise sequence of steps. Understanding this sequence is critical for writing correct state logic.

    Overview

    The locomotion job runs as a Burst-compiled IJobChunk scheduled in the FixedStepSimulationSystemGroup, after the physics step. Each character is processed independently and in parallel.

    The steps execute in this exact order:

    Step Phase Your Code
    1 SolveOverlaps
    2 HandleExternalImpulse
    3 Integrate OnIntegrateState
    4 Collide-and-Slide (loop) OnResolveStateCollisions
    5 SnapToGround
    6 ComputeInheritedDisplacement
    7 EvaluateTransitions EvaluateTransitions

    There is no OnUpdateState method. The integration and collision resolution phases together form the "update" of your state. Transitions are evaluated once, at the very end, after all physics is resolved.

    Step-by-Step Breakdown

    1. SolveOverlaps

    Before any movement happens, the system checks whether the character's collider is currently overlapping with any other collider. This can happen when:

    • A moving platform pushes the character into a wall
    • Another character occupies the same space
    • The character was teleported into geometry

    The overlap solver attempts to push the character out of the overlap. If it cannot (e.g., the character is pinched between two moving surfaces), it tracks the overlap state:

    OverlapState Meaning
    None No overlap detected
    Push Character was pushed out successfully
    Pinch Character is squeezed but not fully trapped
    Crush Character is fully embedded with no escape

    You can read ctx.BaseContext.OverlapState in your state logic to react to crushing situations (e.g., kill the character).

    2. HandleExternalImpulse

    The CharacterContactSystem runs before the locomotion job and collects velocity impulses from external sources, such as:

    • Dynamic rigid bodies that have collided with the character
    • Other characters pushing against this one
    • Explosions or force fields

    These impulses are applied to the character's velocity before your state logic runs, so your OnIntegrateState sees the updated velocity. This is why you should not clear velocity unconditionally at the start of integration unless you intend to discard external forces.

    3. Integrate (OnIntegrateState)

    This is where your state logic computes velocities and displacements for the current frame. The system calls your state's OnIntegrateState method.

    In this step, you should:

    • Compute acceleration from input (movement direction, jump force)
    • Apply gravity acceleration
    • Call baseContext.UpdateVelocitiesAndComputeDisplacements() to convert velocities into displacements

    What you should NOT do here:

    • Move the character's position directly (that happens in the collision step)
    • Add platform velocity (the ComputeInheritedDisplacement step handles that automatically)
    • Read contact information from this frame (contacts haven't been resolved yet; you see last frame's contacts)

    The Kinematic Equation

    UpdateVelocitiesAndComputeDisplacements applies the standard kinematic equations:

    displacement += velocity * dt + 0.5 * acceleration * dt^2
velocity += acceleration * dt

    This is done separately for horizontal and vertical components, allowing different behavior for each (e.g., gravity only affects vertical, movement input only affects horizontal).

    4. Collide-and-Slide (OnResolveStateCollisions)

    After integration computes the desired displacement, the character needs to actually move through the world while respecting colliders. This is the collide-and-slide loop.

    The system calls your state's OnResolveStateCollisions, which should call baseContext.Move(). The Move() method:

    1. Projects the displacement along the current contact surface (if any)
    2. Casts the collider forward by the projected displacement
    3. If a collision is detected, updates the contact information and reduces the remaining displacement
    4. Returns control to the system

    The system re-invokes OnResolveStateCollisions as long as there is remaining displacement, up to CharacterPhysicsProfile.MaxBounce iterations. This multi-bounce approach allows the character to slide along walls, step over obstacles, and navigate complex geometry.

    Contact-Dependent Behavior in Move()

    The Move() method applies different displacement corrections depending on the contact type:

    Contact Type Behavior
    None Displacement is applied as-is (free movement in air)
    Ground Vertical displacement is cancelled; horizontal displacement follows the ground slope
    Wall Displacement is projected along the wall surface; step detection is attempted
    Ceiling Vertical displacement is projected along the ceiling; horizontal displacement is constrained

    Step Detection

    When a wall contact is detected, the system checks whether it could be a step:

    1. The slope must exceed MaxSlope
    2. The character must be moving into the slope
    3. The contact height must be below MaxStepHeight

    If all conditions are met, PhysicsOperations.TryStep attempts to move the character up and over the step.

    5. SnapToGround

    After all collision resolution is complete, the system performs a downward raycast to detect the ground surface below the character. This ensures the character stays grounded when walking down slopes or stairs, rather than becoming airborne for a frame.

    The snap distance is computed by your context's ComputeGroundSnappingDistance method. The default implementation accounts for:

    • The maximum step height
    • The current slope angle
    • The character's horizontal speed (faster characters need to check further down)
    • Platform vertical velocity (if riding a descending platform)

    If a ground surface is found within the snap distance, the character's position is adjusted downward and the contact information is updated.

    6. ComputeInheritedDisplacement

    If the character is standing on a moving platform (a kinematic body with the Platform tag), this step computes the velocity and rotation the character inherits from the platform.

    The inherited values are stored in:

    • baseContext.PlatformHorizontalVelocity
    • baseContext.PlatformVerticalVelocity
    • baseContext.InheritedRotation

    These are available to your state logic in the next frame's OnIntegrateState. The actual position adjustment from platform movement is applied automatically, which is why you should never add platform velocity manually in your state logic.

    7. EvaluateTransitions

    The final step calls your state's EvaluateTransitions method once. By this point:

    • The character has moved to its final position
    • All contacts are resolved and up-to-date
    • Platform inheritance is computed

    Your transition logic can inspect the full post-physics state to decide whether to change states. The convention is to chain transition checks with early returns:

    internal static void EvaluateTransitions(ref MyContext ctx, out MyStates newState)
{
    if (IdleToJumping(ref ctx, out newState)) return;
    if (IdleToMoving(ref ctx, out newState)) return;
    if (IdleToFalling(ref ctx, out newState)) return;
}

    The order of checks determines priority. The first matching transition wins.

    Important

    Transition helper methods must call OnExitState on the current state and OnEnterState on the new state before returning true. This ensures proper state initialization.
